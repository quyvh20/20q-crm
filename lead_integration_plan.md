# Third-Party Lead Integration Platform — Phased Plan (L0–L7)

> Status tracker. Each phase is independently deployable (Railway backend first, Pages frontend second). New tables/columns ship as **both** numbered `migrations/*.sql` (fresh installs) **and** idempotent `main.go` boot guards (prod — golang-migrate is dead-dirty at v2). Every new env var needs a `Config` struct field **and** an explicit `viper.BindEnv` in `LoadConfig` (the `PADDLE_WEBHOOK_SECRET` miss is the cautionary tale). Public callback URLs are config-derived, never `c.Request.Host`-derived.
>
> Scope: get leads INTO the CRM from third-party platforms (Facebook Lead Ads, Google Ads lead forms, web forms, Zapier/Make, any custom system) through one shared ingestion pipeline, plus a public **Lead Capture API** ("the API connection port"). Outbound sync (pushing CRM data to other tools) is out of scope — the automation engine's `send_webhook` action already covers the basic case.
>
> This plan was adversarially reviewed pre-build (5-lens multi-agent pass, 45 findings applied): the notable design corrections vs. the first draft are called out inline as **[review]**.

## Why

Leads are the app's supply line, and today there is exactly **one** way for an external system to create one: `POST /api/webhooks/inbound/:org_token` — a single hardcoded per-org endpoint with acknowledged v1 debt:

- **Raw-SQL upsert bypasses RecordService** (`automation/handlers.go:849`): no field validation, no FLS, no owner assignment (own-scoped reps never see the leads), no record number, no search indexing, `custom_fields` wholesale-replaced on update, INSERT errors unchecked (a dup-email race returns 200 with a phantom contact id).
- **One token+secret per org** (`automation_workflow_org_tokens`, OrgID is the PK): all external sources share one credential; compromising one burns all; the token rides in the URL (leaks into proxy logs); `WEBHOOK_SKIP_SIGNATURE=true` globally disables HMAC for every org.
- **Email-required, hardcoded field map**: Facebook leads are often phone-only; custom questions can't be mapped; nothing is configurable per source.
- **No dedupe strategy** beyond exact case-sensitive email; no phone matching; `contactUseCase.Create` turns the unique-index violation into an HTTP 500.
- **No provider connections**: Google OAuth tokens are used once for login and discarded — there is no storage, encryption, refresh, or health machinery for any third-party credential (FB page tokens, TikTok advertiser tokens).
- **No attribution**: leads arrive with no source, campaign, or UTM data, so "which channel produces revenue" is unanswerable in P9 reports.
- **No monitoring**: a dead webhook fails silently; there is no event log, no health status, no notification.

The platform landscape (researched + independently re-verified 2026-07): Google Ads lead form webhooks require **zero** platform gating (the advertiser pastes your URL+key into the Google Ads UI). Meta/Facebook Lead Ads requires a Meta app + Advanced Access App Review + Business Verification + Access Verification — realistically **several weeks** of business-process lead time that must run in parallel with engineering, and Zapier/Make can bridge FB/TikTok/LinkedIn leads into a generic capture API from day one (note: Zapier's webhook app is a paid-plan feature; **Make's free tier is the zero-cost bridge**). LinkedIn's native Lead Sync API is verification-gated and version-churny — the relay recipe is the documented path. So the build order is value-first: **generic capture API → Google Ads → form embeds → Facebook native → TikTok/Instagram**.

## Architecture

### New package: `internal/integrations`

Follows the automation package's inversion pattern: depends only on `domain` ports it declares (or that exist), receives `RecordService`, `PermissionUseCase` (authorizer/audit), notification usecase, and Redis injected from `cmd/server/main.go`. `integrations` may import `internal/ai` later (lead scoring) but nothing imports `integrations` except `main`. It owns its route registration — **but note [review]:** the `autoHandler.RegisterRoutes` precedent (`main.go:1353`) receives only the plain session `AuthMiddleware`, which would silently skip `RequireTwoFactorSatisfied` and reject PATs. Integrations management routes must receive the **protected-group-equivalent stack** (`AuthMiddlewareWithAPITokens` + `RequireTwoFactorSatisfied`) injected from main.go, or register through the delivery protected group — otherwise invariant 8's "PAT scope works automatically" is false and the workspace 2FA policy grows a bypass.

### Data model (numbered migrations + main.go boot guards, org_id on everything)

- **`lead_sources`** — one row per inbound channel instance. `id uuid PK`, `org_id`, `kind` (`api` | `google_ads` | `form_embed` | `facebook_form` | `tiktok_form` — CHECK constraint), `name`, `connection_id uuid NULL`, `token_hash varchar(64) UNIQUE NULL` + `token_prefix varchar(16)` (api keys, SHA-256 like PATs), `public_token varchar(64) UNIQUE NULL` (URL-addressed kinds), `secret_hash varchar(64) NULL` (google_key etc.), `target_slug` default `'contact'`, `field_map JSONB` default `'{}'`, `match_fields JSONB` default `'["email"]'` (dedupe order, Zoho pattern), `update_policy` (`fill_blank_only` default | `overwrite` | `create_only`) **[review]**, `default_owner_id uuid NULL`, `owner_pool JSONB NULL`, `config JSONB`, `status` (`active`|`disabled`|`error`), `consecutive_failures int`, `last_used_at`, `created_by`, `created_at`, `disabled_at NULL`, `deleted_at NULL` (**soft-delete** — hard delete would orphan the event ledger L6 depends on; delete-confirm warns when `last_used_at` is recent because the external side keeps POSTing) **[review]**.
- **`integration_connections`** (L5) — one row per OAuth'd provider account. `id`, `org_id`, `provider`, `external_account_id`, `external_account_label`, `encrypted_credentials TEXT` (envelope format below), `key_version int`, `webhook_secret_encrypted TEXT NULL`, `status` (`connected`|`degraded`|`error`|`revoked`|`disconnected`), `cursor JSONB`, `config JSONB`, `last_synced_at`, `last_error text`, `consecutive_failures int`, `created_by`, timestamps. `UNIQUE(org_id, provider, external_account_id)` plus a **partial** unique claim index `(provider, external_account_id) WHERE status IN ('connected','degraded','error')` **[review]** — an *active* claim is exclusive (app-level webhooks need unambiguous page→org routing: this is a **product choice we own**, not a Meta constraint — Meta happily feeds multiple CRMs per page), but disconnect frees the claim so page transfers and same-org reconnects work. Reconnect by the same org **upserts** (refresh credentials, reset status/failures); a cross-org attempt on an actively-claimed page returns a friendly "already connected to another workspace" error (never a raw constraint violation, never naming the workspace).
- **`integration_events`** — the inbound ledger AND the async work queue; the state machine is explicit **[review]**:
  - Columns: `id`, `org_id`, `source_id NULL`, `connection_id NULL`, `provider_event_id text NULL`, `status`, `claimed_at NULL`, `attempts int`, `raw_payload JSONB` (verbatim — mapping drift can never lose data), `context JSONB`, `result_slug`, `result_record_id uuid NULL`, `outcome` (`created`|`updated`|NULL), `error text`, `created_at`, `processed_at`.
  - Statuses: `pending` (async, claimable) | `processing` | `processed` | `duplicate` | `failed` | `quarantined` | `test`. **Sync channels (L1–L4) insert directly as `processing`** — never claimable; only provider-webhook channels (L5+) insert `pending`. The async worker claims `pending → processing` (`FOR UPDATE SKIP LOCKED`, sets `claimed_at`, `attempts++`); a stale-claim reaper flips abandoned `processing` rows (claimed_at older than N min AND `result_record_id IS NULL`) back to `pending` — deploy/crash mid-fetch must not double-write, so **`result_record_id` is persisted before any side effect fires** and the retry path re-runs the match first. Partial index on `status='pending'` for the poll loop.
  - Dedupe: **two** partial unique indexes **[review — the single `(source_id, …)` index silently failed for FB, where `source_id` is NULL at webhook receipt and Postgres treats NULLs as distinct]**: `(source_id, provider_event_id) WHERE source_id IS NOT NULL AND provider_event_id IS NOT NULL` and `(connection_id, provider_event_id) WHERE connection_id IS NOT NULL AND provider_event_id IS NOT NULL`. Provider webhooks insert connection-scoped. Verified by a test inserting the same `leadgen_id` twice with NULL `source_id`.
  - Retention: pruned to 90 days (L6); PII redacted earlier on contact deletion (L6).
- **`integration_oauth_states`** (L5) — server-side single-use OAuth state: `state_hash UNIQUE`, `org_id`, `user_id`, `provider`, `return_to`, `expires_at` (10 min), `consumed_at NULL`. **Never encode org/user IDs in the state parameter itself** — that's the same capture-vulnerability class the U4 review killed.
- **`integration_pending_connections`** (L5) **[review]** — custody for the exchanged token between callback and account selection: `id`, `org_id`, `user_id`, `provider`, `encrypted_token`, `candidate_accounts JSONB`, `selection_token_hash UNIQUE`, `expires_at` (10 min), `consumed_at`. The account picker never sees the provider token — the FE gets only the candidate list + a single-use selection token; the select-account POST verifies the caller matches the pending row's org/user before promoting to `integration_connections`.

### The shared ingest pipeline (`LeadIngestService`) — every channel funnels here

`Ingest(ctx, source, RawLead{fields, context, consent, providerEventID, isTest, suppressAutomation})`:

1. **Dedupe the delivery** — event-row insert with `ON CONFLICT DO NOTHING` on the applicable partial index; conflict → return prior result (`duplicate`), no side effects. Backstop for FB's 36-hour retries and Google's at-least-once delivery.
2. **Map + validate fields — strict from day one [review].** The first draft deferred validation to L2; that left L1's trusted write open to field injection. Corrected: every phase, only keys that survive the source's `field_map` AND exist in the registry schema for `target_slug` are written. **Ownership and relation keys (`owner_user_id`, `company`, any relation-typed field) are never acceptable from a caller payload and never valid `field_map` targets** — owner comes exclusively from source routing config (validated as an active org member at save time and at stamp time), relations only via explicit later features. Unmapped/rejected keys are recorded on the event row (`quarantined_fields`), never silently dropped and never written. In L1 the `field_map` is an auto-generated identity map over the target's schema keys (semantically: `field_map={}` **permanently** means "identity over schema-valid keys" — L2's engine must not reinterpret empty maps, so live L1 sources behave identically the day L2 deploys **[review]**).
3. **Match** — normalize email (trim/lowercase) and phone (E.164 best-effort, L2); look up by the source's `match_fields` order via new `ContactRepository.FindByNormalizedEmail` (exact `LOWER(email)` lookup — the existing code has none; `List(Q:email)` is full-text and can mismatch; new index `CREATE INDEX IF NOT EXISTS idx_contacts_org_lower_email ON contacts(org_id, LOWER(email)) WHERE deleted_at IS NULL`). Deterministic pick when case-variant duplicates pre-exist: `ORDER BY created_at ASC LIMIT 1` **[review]**.
4. **Write through RecordService** with an **upsert loop [review]**: no match → `Create`; on a unique-violation error (concurrent first-time deliveries racing — the exact bug the legacy webhook has) re-match and fall through to update. Match → apply the source's `update_policy`: `fill_blank_only` (default — a rep's manual corrections are never clobbered by a form resubmission; attribution fields follow their own first/last-touch rule), `overwrite`, or `create_only` (skip update entirely). **`first_name` synthesis happens only on the create branch** (provider full-name split, else email local-part, else `"Lead"`) — a synthesized name must never overwrite a real one on update **[review]**.
   - **Actor decision (explicit, documented):** ingestion runs as the **org-level system actor** — a fresh `context.Background()`-derived context with **no** `domain.Caller` (trusted in-process call) and **no** `MarkHTTPTransport` stamp (avoids the U0.10 `warnCallerlessHTTP` alarm). Audit lands as `ActorID uuid.Nil` ("System") with the source id in the audit payload. We deliberately do **not** run as the source creator (P8 pattern): a lead pipe is org infrastructure and must survive the creator's departure — the PAT-dies-with-membership failure mode. Two consequences, both handled: (a) FLS/OLS don't constrain the write, so step 2's strict schema allowlist + the ownership/relation blacklist are **load-bearing security controls**, and **source save/edit authorizes the *configuring admin's* OLS create+edit on `target_slug` with their real caller** **[review]** — `integrations.manage` alone must not become an OLS bypass onto sensitive objects; (b) `target_slug` is restricted to **system objects backed by adapters** (contact now; company/deal if ever useful) — the custom-object path sets `created_by = userID` with an FK to `users(id)`, so a `uuid.Nil` system write is an FK violation; custom-object targets are **deferred** until that path accepts a NULL creator **[review]**.
5. **Stamp attribution** — system-stamped origin (`integration:{kind}` — callers cannot forge it) + parsed `context` (UTM/gclid/fbclid from `page_url`, referrer, IP) into the L2 attribution fields; first-touch on create, last-touch keys on update.
6. **Fire events with context** — the L0 plumbing carries `WithWriteSource` (so `trigger.source` names the channel) and `WithAutomationSuppressed` (test leads, backfill) into the emit sites; anything fire-and-forget runs on `context.Background()` (gin-cancellation trap).
7. **Record outcome** — event row → `processed`, `result_record_id`, `outcome` (persisted **before** step 6's side effects, see the state machine) → response.

Test leads (`RawLead.TestOrigin`) traverse the FULL pipeline but are **silenced** and badge the event `test`. **[review]** Suppression is a sabotage vector if invisible — a leaked google_key could submit real-looking leads flagged `is_test` that never page sales — so provider-supplied `is_test` volume is surfaced in L6 health (spike alert), while the admin's own "Send test lead" button uses a server-side flag that is never caller-controlled. `TestOrigin` (`admin` | `provider`) exists precisely to keep those two apart: L2's identity guards apply only to the admin origin, since an L3 provider test carries a real payload we did not build.

**L2 decided the silence question (built — see L2.6):** L0's suppression is quiet, not silent, and this plan's stated mitigation ("deleting the test record cancels its timers") turned out to be **false in this codebase** — `contact_deleted` is emitted only by `RecordService.Delete`, while the UI deletes via `DELETE /api/contacts/:id` → `contactUseCase.Delete`, which emits nothing. A timer armed by a test lead therefore outlives its own record with no UI-reachable cancellation. So L2 added `domain.WithAutomationSilenced`.

### L0 — the RecordService event-context plumbing (own pre-slice) — **DONE (unpushed)**

The first draft called this "a small extension"; it is a cross-cutting change to the event-emission path of **every existing write** and must not share a PR with a new public endpoint.

**Built:** `domain.WithWriteSource(ctx, s)` / `WriteSourceFromContext` (defaults to `DefaultWriteSource = "crm_ui"`) and `domain.WithAutomationSuppressed(ctx)` / `IsAutomationSuppressed` in `caller.go`, plus the shared `AutomationSuppressedPayloadKey = "_suppressed"` const (it lives in `domain` because `usecase` must not import `automation` — the emitter is injected — and because the emitters detach onto `context.Background()`, so the payload is the only channel to the engine). `ctx` threaded into all **three RecordService emitters** — `fireLifecycleEvent` + `fireStageChanged` (`record_service_system.go`) and the custom-object `fireEvent` (`record_service.go`) — via two shared helpers (`triggerMeta` / `markSuppressed`), 13 call sites. `engine.go` gained `isEnrollmentSuppressed`, placed beside the existing `_internal_update` loop guard so run creation is skipped while `materializeDateFieldTimers` (called after it, same goroutine) still runs.

**Scope correction [review]:** the count of "three" is right only for the *uniform write path*. A grep finds **eight** live `crm_ui` literals: the five others are legacy `delivery/http` handlers (`contact_handler.go:212/:272`, `deal_handler.go:191/:284`, `custom_object_handler.go:246`) that call their own usecases, not RecordService, and build their own payloads. They are deliberately **out of L0's scope** — leaving them untouched is what preserves zero behavior change, they are user-facing routes that never set the flag, and every server-side bulk writer (L1's ingest, L7.4's legacy shim) goes through RecordService. The constraint this creates is documented on `WithAutomationSuppressed`: **a new server-side writer must route through RecordService to inherit suppression.**

**Semantic correction [review]:** suppression is **quiet, not silent** — a suppressed write still arms `date_field` timers, which later fire real runs (the timer's payload is rebuilt without the flag). That is correct for backfill (a backfilled lead's close-date reminder should still arm; the enrollment storm is what's being prevented) but it meant **L2's "Send test lead" would not be fully silent**. That decision was deferred to L2, where the button is actually built.

**→ L2 RESOLVED IT (built): `domain.WithAutomationSilenced` now exists.** Three findings from L2's design pass, all verified against the code, killed the "just reuse suppression" option and shaped the fix:
1. **The mitigation named above ("deleting the test record cancels its timers") is FALSE here.** `contact_deleted` is emitted only from `RecordService.Delete` (`record_service_system.go:179`); the UI deletes through `DELETE /api/contacts/:id` → `ContactHandler.Delete` → `contactUseCase.Delete`, which emits **nothing**. A timer armed on a test lead is therefore uncancellable through the product and fires days later against a record that no longer exists.
2. **The obvious fix is a no-op that reads like a fix.** `fireTimerRun` (`timers.go:250`) builds its run straight from the stored timer payload and calls `CreateRun` directly — it consults no suppression predicate. Propagating a flag into `timerPayload` changes nothing. Silence can only be honored by **not arming**. Pinned by `TestFireTimerRun_IgnoresSuppressionFlags`.
3. **The guard wraps the ARM only** (`datefield_timers.go`, immediately before `timerPayload`) — never the function. Both branches above it are *cancels*, and a cancel only disarms; silencing one would strand the exact timer silence exists to prevent. Pinned by `TestMaterializeDateFieldTimers_SilencedDeleteStillCancels`.

Composition: silence **derives** suppression (`IsAutomationSuppressed` returns `v || IsAutomationSilenced(ctx)`), so a caller cannot ask for silence and get an enrollment storm. The two failures are not equally expensive — forgetting silence costs one stray timer, forgetting suppression costs a stampede — so the expensive one is made unrepresentable rather than merely remembered. `markSuppressed` → `markAutomationFlags` stamps both keys with two independent `if`s (an `else if` would emit silenced-but-not-suppressed and enroll every test lead, since the engine's guard reads only `_suppressed`).

**Verified:** build + vet clean; gofmt-clean modulo the repo's CRLF working-tree artifact (`core.autocrlf=true`; the flagged files have *no* real drift — verified with a CR-insensitive gofmt diff — so `gofmt -w` stays banned). Full **Docker-backed** suite green with **zero skips**: `automation` 438s, `usecase`, `domain`. Each of the three emitters is pinned by BOTH a suppressed case and an unsuppressed control asserting the default source and the **absence** of the suppression key (the control is the load-bearing half: a positive-only assertion cannot catch an always-on flag). Two Docker-gated engine tests prove the contract end-to-end against real Postgres — a suppressed write arms the date_field timer and creates **zero** runs; the unsuppressed control creates one, proving the fixture could enroll. Test strength was mutation-proven, not assumed: reverting `fireStageChanged` to its pre-L0 body and making `fireEvent` stamp the flag unconditionally each leave the pre-review suite green, and each now fails.

**Done when:** ~~log-verified in prod~~ — **[review]** no log line emits `trigger.source`, so the invariant isn't greppable; the real verification surface is `workflow_runs.trigger_context->'trigger'->>'source'` (also visible in Run History), which must read `crm_ui` for UI writes after deploy.

### Security invariants (checked in every phase's review)

1. All signature verification is HMAC-SHA256 over the **raw body bytes**, compared with `hmac.Equal`. No global skip flag — `WEBHOOK_SKIP_SIGNATURE` is not extended to any new endpoint.
2. Inbound credentials (API keys, google_key) are stored **hashed** (SHA-256, PAT pattern), shown once via SecretReveal, rotatable in one click, carried in the `Authorization` header where the caller controls headers.
3. Provider OAuth tokens are **AES-256-GCM envelope-encrypted**: per-connection random 32-byte DEK wrapping the token blob; DEK wrapped by a KEK from `INTEGRATION_ENC_KEY`; `key_version` beside every ciphertext keyed to a named KEK keyring. **[review]** This is **net-new engineering** — `two_factor_crypto.go` is pattern inspiration only (single static key, no envelope/keyring exists anywhere in the codebase), and there is **no `JWT_SECRET` fallback**: the KEK falling back to an unrelated rotatable signing secret would silently invalidate every stored credential on a routine JWT rotation without bumping `key_version` (unrecoverable data loss). Boot fails loud if provider features are configured without `INTEGRATION_ENC_KEY`; a startup canary decrypt catches KEK mismatches immediately. Tokens are **never** returned to the frontend.
4. OAuth state is random, server-stored, single-use, 10-minute expiring, org/user-bound server-side; PKCE S256 wherever supported. Callback handlers resolve the org **from the state row**. The account-picker interstitial uses the `integration_pending_connections` custody flow — the exchanged token never rides through the browser.
5. Public endpoints: registered outside AuthMiddleware, `io.LimitReader` body caps (256KB capture / 1MB provider webhooks), per-source rate limits, fast-ack + async for provider webhooks. **[review]** The limiter is a **new build** (Redis fixed-window per source key; the existing `RateLimitByIP` is auth-prefixed/IP-keyed and not reusable as-is) and on public *write* endpoints it **fails closed to a strict in-process token bucket** when Redis is unavailable — never open: each request can create records, enroll workflows, and send billable Resend email, so an unthrottled window is a cost-amplification bomb. A DB-enforced per-source daily creation cap bounds amplification even if both limiters misbehave. Test-lead and batch endpoints are limited too.
6. Every table row and every query is org-scoped; provider-webhook routing resolves connections by `(provider, external_account_id)` **filtered to active status** **[review]** — events for revoked/disconnected pages are quarantined, never written into the last workspace that held the page.
7. Public URLs (capture URLs, OAuth redirect URIs, webhook URLs) derive from a new `PUBLIC_API_BASE_URL` config (BindEnv!) defaulting to the current Railway host — never `c.Request.Host`.
8. New capability `integrations.manage` gates all management surfaces (Go `domain.AllCapabilities` + `CapabilityCatalog` **and** FE `ALL_CAPABILITIES`/`CAPABILITY_LABELS` in lockstep) — and the management routes accept PATs + enforce 2FA via the protected-equivalent middleware stack (see package note).
9. Caller-supplied `Idempotency-Key` is namespaced per source **and** the event log warns when a duplicate collapses a payload that differs materially from the stored one **[review]** (a shared/compromised key must not become a silent lead-suppression primitive).

## L1 — Lead Capture API (the universal port) + Integrations shell

**Problem:** No sane way for any third-party system to push a lead in. This one endpoint is simultaneously the Make/Zapier integration, the LinkedIn workaround, the custom-website story, and the backbone every later channel reuses. Depends on L0.

### L1.1 — Backend capture path (curl-verifiable, ships alone)

> **Corrections forced by the pre-build recon (6 exact-contract readers) — the spec below was partly unimplementable as written:**
>
> 1. **The upsert loop was impossible.** `contact_usecase.go:87` did `if err := contactRepo.Create(...); err != nil { return nil, domain.ErrInternal }` — the pgconn 23505 was discarded, and `domain.AppError` has no `Unwrap()`, so a duplicate email was indistinguishable from a dead connection. **Prerequisite landed:** `domain.ErrContactEmailExists` (409) + `isContactEmailConflict` matching SQLSTATE 23505 **and** the constraint name `idx_contacts_org_email` (a constraint-blind check would read any future unique index on contacts as an email dup and update the wrong row). This also fixes the "dup email → HTTP 500" debt named in this plan's Why. `gorm.ErrDuplicatedKey` is unusable — the shared DB is opened without `TranslateError`, so that sentinel never fires. **The plan's L1.1 Files list omitted `contact_usecase.go`; it is required.**
> 2. **Case-sensitivity split the design.** The unique index is on raw `(org_id, email)` — case-SENSITIVE — while the plan matches on `LOWER(email)`. So `John@X.com` and `john@x.com` both insert, no violation fires, and the upsert loop never engages. **Fix:** ingestion normalizes (trim + lowercase) email *before the write*, so the match key and the unique index agree and the loop actually guards the race. `FindByNormalizedEmail` is **deliberately unscoped** (org filter only): dedupe is not a read of the caller's data — scoping it would turn every invisible contact into a duplicate. Its `ORDER BY created_at ASC` is load-bearing, since pre-existing case-variant twins are legal today. The new `LOWER(email)` index stays **non-unique** (promoting it would fail silently on any org that already has case-variant dupes — see the `main.go:718-745` unique-promotion ritual).
> 3. **Custom fields are quarantined in L1.1 (scope cut).** `validateSystemCustomFields` early-returns only when *zero* custom keys are present; send **one** and `ValidateFields` demands **every** required custom field in the org — so a single mapped custom key 400s the entire lead in any org with an unrelated required field. L1.1's allowlist is therefore **schema-derived native writable keys only** (`first_name`, `last_name`, `email`, `phone`), with everything else — including legitimate org custom fields — recorded verbatim on the event row as quarantined rather than written. L2's mapping engine owns custom fields and must solve the required-field trap (call `ValidateValue` per key; never let `ValidateFields`' required-loop run over an ingest payload).
> 4. **Email is required on the L1.1 capture payload** (422 without it) — not a preference. `LOWER(NULL)` is NULL and the unique index is partial on `email IS NOT NULL`, so a phone-only lead can neither match nor conflict: it would insert unboundedly. This is the plan's own "API sources may declare email required" line, promoted to a hard L1.1 constraint until L2 adds phone matching.
> 5. **Audit source attribution** — RecordService's `auditCreate` builds `AuditEntry` privately with no metadata hook, so the source id cannot ride into its audit row. ## L2 — Field mapping, dedupe & routing hardening

**Problem:** L1 maps identity-over-schema only, dedupes on email only, and parks attribution in the event log. Real sources rename fields, send phone-only leads, and need routing.

**Build:**
1. **Mapping engine:** `field_map` entries `{source_key → {target_key, transform?: none|split_name|to_number|to_date}}`. Save-time validation (registry schema + the ownership/relation blacklist, already enforced since L1); per-lead failures are per-field — the lead is still accepted, bad fields quarantine onto the event row. `field_map={}` keeps meaning identity-over-schema (L1 sources behave identically before/after this deploy — explicit Done-when item).
2. **Mapping UI:** table `source field | target field (registry dropdown) | status`; identity fields locked; "captured but unmapped" rows get one-click "create custom field & map" (admin-gated).
3. **Attribution fields [review — collision semantics defined]:** on first source creation, seed org custom field defs: `lead_source`, `lead_source_detail`, `utm_source`, `utm_medium`, `utm_campaign`, `utm_term`, `utm_content`, `referrer_url`, `gclid`, `fbclid` — **all plain text** (a seeded *select* would freeze its options and 400 entire contact writes when L7 adds new kinds; `fieldvalidate` rejects unknown select values). Collision rule: if the org already has a same-key field, adopt it only when type-compatible (text-like); otherwise seed under the reserved `crm_` prefix (`crm_lead_source`) and surface the conflict in the mapping UI. Ingest parses `context.page_url` server-side; first-touch on create, last-touch on update. Ordinary custom fields → reportable in P9 and usable in workflow conditions immediately.
4. **Phone dedupe:** E.164 best-effort normalization (org default region), new nullable `contacts.phone_normalized` (boot guard + index `(org_id, phone_normalized)`); `match_fields` may be `["email","phone"]`, first match wins.
5. **Owner routing (advanced) — DONE (verified live).** `owner_pool` round-robin + the owner UI (`default_owner_id` landed in L1's BACKEND only — no picker was ever built, so every source created through the UI produced unowned contacts, verbatim the failure this plan opens with). Decisions (3 designs + 4 hazard angles, 40 hazards):
   - **Cursor = a monotonic TICKET counter, never an index.** `lead_sources.owner_cursor BIGINT`, advanced by ONE statement that also returns the pool: `UPDATE … SET owner_cursor = owner_cursor + 1 WHERE … jsonb_array_length(owner_pool) > 0 RETURNING owner_cursor - 1, owner_pool` (the `record_number_repository.go` precedent). An index is undefined the moment the pool shrinks and every implementation then clamps — the clamp is where the skew hides. Postgres row-locks for the statement's duration, so concurrent leads serialize; **mutation-proven**: swapping it for a Go read-modify-write makes 40 concurrent leads collide on tickets. Nothing is held across the RecordService write. BIGINT because a negative ticket would panic the slice index.
   - **The columns are NOT on the GORM structs.** Two independent traps: `UpdateSource` is `db.Save`, so a mapped cursor would let an admin renaming a source rewind the rotation to its page-load value; and the boot-guard loop logs-and-boots, so a mapped `owner_pool` whose ALTER failed would land in `FindSourceByTokenHash`'s SELECT and 500 **every capture request in every org**. Unmapped, a failed guard breaks one advisory statement and routing degrades to the fallback.
   - **Filter live members FIRST, then modulo.** Ranging the liveness MAP instead of the pool SLICE makes rotation a random draw (Go randomizes map iteration) and the cursor decorative — and a distribution test passes either way, so the tests assert exact SEQUENCES. Modulo-then-skip gives one member a double share: [A,B,C] with B suspended must be A,C,A,C, not A,C,C.
   - **Three-rung ladder, never a failed lead:** live pool member → `default_owner_id` (re-verified live, and only on pool-configured sources so L1 semantics are byte-identical for everyone else) → written UNOWNED and loud in four places (ledger note, `slog.Error`, a `warnings` key on the capture response, and a server-computed `owner_pool_inactive` badge). A 4xx to a one-shot Make/Facebook delivery usually means the lead exists nowhere; an unowned contact at least exists. Stamping a suspended member is worse than unowned — it looks assigned, so nobody triages it.
   - **Liveness failure fails OPEN** — the opposite polarity to the daily cap, deliberately: "a liveness result we could not obtain is not evidence the pool is dead." Treating an error as an empty result would unown real leads on a DB blip.
   - **The ticket is taken lazily at the create branch only.** An update-path lead takes none (a source that is 90% resubmissions would otherwise burn 90% of its turns and the rotation would degenerate while looking healthy); a test lead PEEKS; a failed write burns its ticket permanently (never decremented — a "rollback" reintroduces the race); a retry REUSES `integration_events.assigned_owner_id`, without which a form where every other lead fails hands one rep 100% of contacts with a green ledger.
   - **Strict rotation, diverging from automation's `assign_user`,** whose `round_robin` is actually least-loaded (counts all-time owned rows, no membership check). The integrations UI never says the bare words "round robin". **Two live defects found in that executor and NOT fixed here (separate task):** a departed member's frozen count becomes the permanent minimum so every assignment routes to the person who left; and its `least_loaded` fallback selects from `users WHERE org_id` (the legacy denormalized column) rather than `org_users`, so it can assign to a non-member.
   - **FLS gap closed:** configuring ownership now requires the caller's own `FieldMask.CanWrite("owner_user_id")`, matching `executor_assign.go`. Ingest writes callerless, so `integrations.manage` was otherwise an ownership-assignment primitive.
   - **Offboarding:** removal prunes the member from every pool (in `RevokeUserGrants`' existing transaction, same "a re-invite must not silently restore access" rationale — `org_users` PK is `(user_id, org_id)`, so a re-invite flips the same row back to active) and the reassignment dialog names the affected sources. Suspension deliberately does NOT prune: it is reversible and routing already skips it.
   - **Source creation now REQUIRES a routing choice** (one owner / a rotation / deliberately unassigned behind a warning).
6. **Test-lead flow — DONE (verified live).** `POST /api/integrations/sources/:id/test-lead` (protected + `integrations.manage`, **no request body** — there must be no wire on which a caller could supply an identity or an is_test flag) drives a server-built payload through the REAL pipeline; deep-links the record AND the delivery; event badged `test`. Decisions taken (4 designs + 3 judge lenses + 5 adversarial hazard angles):
   - **Silenced, not suppressed** — see the L0 section above for why the plan's own mitigation was false.
   - **Collision safety is by construction, not by luck.** A test lead runs the real upsert, so if its identity can match a real contact the button edits a stranger's record (and under `overwrite`, destroys it). Four independent layers: (a) identity `crm-test-{source_id}@lead-test.invalid` — RFC 2606 reserved TLD, derived from `source.ID` so **no column and no boot guard**; (b) **no phone, ever** — there is no safe synthetic phone, because `normalizePhone` reduces to digits and `contacts.phone` is free text validated by nothing on the write path ("no *valid* E.164 number is 16 digits" is true and irrelevant: a real contact holding `+1 555 123 4567 ext 8901234` normalizes to 18 digits); (c) query-side — a test lead **ignores `match_fields`** and issues exactly one lookup for its own address, so the pipeline never issues a predicate that *can* return a real row (this is what covers the phone-only source, where `requiresEmail` is false and the email leg never runs); (d) write-side provenance guard at the single line where the pipeline commits to a pre-existing row, placed **before** the `create_only` early-return (create_only writes nothing but still returns the matched id as "your test lead", which the UI deep-links).
   - **Identity asserted against the server-computed constant, never the payload's own email** — the field_map sits in between and can redirect (`split_name` ignores `TargetKey`), blank, or drop it, and a guard comparing the payload to itself passes vacuously in exactly those cases.
   - **Payload reverse-maps the source's own field_map keys.** A fixed `{email, first_name}` payload would take `Apply`'s passthrough, write a contact, and report success while the mapping was never touched — the test would pass hardest on the most broken source. Type-aware: number/select targets are skipped and **named** in `uncovered`, never guessed (a guessed value 400s the whole write and reads as "the test is broken"). `field_map={}` degenerates correctly via the same passthrough real leads take. Refuses (422) rather than guesses where the mapping would redirect the identity.
   - **Stable identity ⇒ click 1 `created`, click 2+ `updated`** — exactly one test contact per source forever (bounded clutter), and click 2 is the only thing that exercises match → update policy. **No `ProviderEventID`**: a stable one would conflict on the dedupe index and replay click 1's result via the `EventStatusTest` arm of the terminal-success switch — returning green having run **none** of the pipeline.
   - **`authorizeTarget` re-checks the clicking admin's own OLS**, with their real caller, before the ledger insert. Ingest writes callerless, and a test click is neither a create nor a target change — the two places that authorized before — so without this, `integrations.manage` alone becomes a standing contact-write primitive. First negative authz test in the package.
   - **Disabled/`error` sources are testable on purpose** (`create → map → test → enable` is the order that protects leads; blocking would switch the button off exactly when someone is diagnosing a broken source). The result panel says the source is rejecting real traffic right now.
   - **No `TouchSourceUsed` for a test lead** — it stamps `last_used_at` *and* resets `consecutive_failures`, so the button would clear L6's error state on evidence from the one layer it never touches. A diagnostic that silences its own alarm is worse than no diagnostic.
   - **`CountCreatedToday` deliberately NOT given an `AND status <> 'test'` filter** — that exclusion is indistinguishable at the SQL layer from the one that would hand L3's *caller-declarable* `is_test` a free daily-cap evasion. Cost of leaving it counted is at most one cap slot per test-contact lifecycle.
   - **The result panel's "what this did not prove" column is load-bearing, not decoration.** The button is in-process: it skips the network, the capture key, the limiter, the daily cap, and phone matching. That bypass is what makes it safe to hand an admin (no credential in the browser), so it is a boundary to publish, not a flaw to engineer away — the dishonesty risk lives entirely in the copy.
   - **No `contacts.is_test` column and no `is_test` custom field** — the `.invalid` address, the name "Test Lead", and the delivery-log badge are the markers; polluting a customer's schema for a diagnostic is a bad trade.
7. **Consent envelope — DONE (verified live).** Accepts `consent:{basis, text, channels, captured_at}` and stores it verbatim on the delivery row. Decisions (3 designs + 4 hazard angles, 35 hazards):
   - **The live defect it fixes was worse than 'not implemented'.** `RawLead.Consent`, `captureRequest.Consent` and `batchItem.Consent` all existed and were threaded into `Ingest` — and read by nothing. The API accepted a consent object from every integrator and silently discarded it. For a compliance-adjacent field that is worse than not accepting it, because the integrator believes it was recorded.
   - **Ledger only — no contact mirror.** A contact custom field would be an ordinary field a workflow, a hand edit, or the AI write path could silently overwrite; evidence anyone can rewrite is not evidence. It also avoids seeding legal-looking fields onto every org's contact schema and being misread as a suppression list.
   - **Written AFTER the record exists**, adjacent to `SetEventAssignedOwner`, never before. Erasure is contact-keyed, so an envelope stored earlier would survive on failed deliveries where `result_record_id IS NULL` and no erasure request could ever reach it. Cost accepted deliberately: a lead that fails to write loses its envelope — consent about someone who never became a contact proves nothing and would be unerasable PII.
   - **Its own column, not folded into `context`.** `context` is caller-supplied and already mutated by the batch path, so a caller posting `context:{consent:{…}}` would be byte-indistinguishable from a stamped record — destroying the provenance that is the column's only value. NULLABLE with no default so NULL (never sent), an object (sent) and a tombstone (erased) stay distinguishable.
   - **UNMAPPED on the GORM struct.** Three sites enumerate every mapped column (`InsertEventDeduped`, `InsertRefusedEvents`, `FinishEvent`). Worse than the owner_pool case: `FinishEvent`'s `db.Save` runs AFTER the out-of-band consent write, so a mapped-but-unset field would blank the envelope on every success — the silent discard this column exists to fix, reintroduced one layer down. Pinned by a DB test.
   - **A NUL byte was a live lead-loss vector.** Go accepts 0x00 inside a string; Postgres rejects it inside jsonb, so one byte from a public endpoint would fail the write and take the delivery with it. `sanitizeForJSONB` strips NULs and invalid UTF-8 from every string in the envelope.
   - **Nothing malformed ever costs the lead.** Junk types, unparseable dates, unknown bases, oversized bodies — all stored with a warning. `captured_at` is echoed ONLY when it genuinely parses (never defaulted to now(), which would fabricate the one fact the record exists to prove), and oversized `text` is dropped WHOLESALE rather than clipped, because a truncated consent statement reads as the wording the subject saw and is not.
   - **An unknown basis is recorded, not rejected.** Real traffic carries TCPA `express_written_consent` and CASL `implied` alongside GDPR's six; refusing an unfamiliar term would lose a real person's lead over a wording difference. The caller's reserved `_crm` key is stripped before our attestation is merged, so a payload cannot forge what we assert about it.
   - **HONESTY: nothing enforces this.** Verified — no send path in the app consults consent (`executor_email` filters nothing). The stored envelope asserts `enforced:false` in the data itself, and the UI says plainly that it is recorded only, is not an opt-out list, and that no email or workflow is filtered by it. A customer believing they have opt-out enforcement they do not have is this feature's real risk.
   - **Erasure ships with it, not after it.** Deleting a contact redacts `raw_payload`, `context` and `consent` on its ledger rows via a nil-tolerant `LeadLedgerRedactor` port on `contactUseCase.Delete`. Redaction writes a TOMBSTONE, never NULL or `{}` — a blanked row would make the ledger falsely assert no consent was ever obtained, a different and false claim from 'it was obtained and erased on request'. New `idx_integration_events_result` so that redaction does not seq-scan the ledger.
   - **Adjacent defect fixed:** `captureResponse` had no `warnings` field, so `IngestResult.Warnings` — documented as 'said to the CALLER, at integration time' — was discarded on the highest-volume public route, including the unowned-lead warning added in L2.5.
   - **Retention stated truthfully:** there is no prune job, so the UI says the envelope is kept until the contact is deleted rather than printing a 90-day promise nothing implements. The prune stays L6.
   - Deferred, explicitly: enforcement/suppression, withdrawal, per-channel opt-out, and the 90-day retention sweep. Found but NOT fixed here (filed separately): the AI `update_contact` path wholesale-replaces `custom_fields`, silently deleting `lead_source`/`utm_*`; and `truncate` in attribution.go is byte-sliced, corrupting multi-byte runes.
8. **Batch — DONE (verified live).** `POST /api/capture/leads/batch`, ≤100 items, per-item envelope, 207 on any failure. The recovery path L3 names. Decisions (3 designs + 4 hazard angles, 36 hazards):
   - **A batch of N costs what N single requests cost, on every bound.** New weighted `RateLimiter.AllowN` (Redis `INCR`→`INCRBY`, and the fresh-key TTL test moves with it) charging BOTH the key and IP buckets. All THREE local count sites patched — the window-ROLLOVER site hardcoded `count = 1` and is the steady-state path for a recurring caller, so a half-patch would charge 1 for 100 on exactly the Redis-outage path the fallback exists for. Charge is TWO-STAGE (1 before the DB lookup, N-1 after the parse) because N is unknowable pre-parse and a single post-lookup charge would reopen the DB-amplifier hole the single endpoint's ordering closes. Over-limit rejects the WHOLE batch: a prefix-accept splits on state the caller cannot see, so the same batch retried stops somewhere different every time. Per-IP ceiling raised to 1200 so shared egress (NAT, Make/Zapier) has headroom that is a chosen number rather than one bought by undercharging.
   - **Per-item `idempotency_key`, REQUIRED, enforced per item.** Without it a retry is a duplicate factory on the shape L2 already ships: a phone-matching source whose number sits on ≥2 contacts reports ambiguity, refuses to merge, and creates — which raises the match count, so the next retry is ambiguous again. Never converges, and the phone index cannot be UNIQUE. Enforced per ITEM, never as a batch rejection, because a 4xx is terminal for Make/Zapier and one blank id would mean the recovery run does not happen at all. A batch-level `Idempotency-Key` header is a hard 400. Intra-batch duplicates: first wins, later ones refused `retryable` — today the second would silently receive the FIRST's record id, a different person.
   - **Daily-cap headroom decremented ONLY on `!Duplicate && outcome==created`.** The sharpest bug available here: a replayed delivery returns the prior outcome — literally `created` — while the ledger insert wrote nothing, so counting it makes a retry batch spend its cap on rows it did not write and then refuse the genuinely-new leads in its tail, until UTC midnight. That is precisely the recovery shape. Re-read every 25 items so concurrent batches converge.
   - **`daily_cap` was dead.** It defaulted to 0 and no UI ever wrote it, so the branch guarding it had never executed in production — the plan's named backstop was absent. New sources default to **1000/day** and the number is now editable (`*int`, so an explicit 0 CLEARS it; it was one-way before).
   - **Batch deliveries are SUPPRESSED by default** (`WithAutomationSuppressed`, never Silenced — these are real people, so their own date_field reminders should still arm), overridable only by a per-source, UI-visible `batch_enroll_automation` column. NO wire flag: a caller-settable suppression flag would let a leaked key file 100 real leads that land correctly owned and attributed, look ordinary in the ledger, and page nobody. `DeliveryMode` is set from the ROUTE, never the body.
   - **The reaper is a prerequisite, not a nicety.** A delivery whose process died mid-write stays `processing`, and the replay switch answers 409 to every retry — turning the Idempotency-Key into the thing that makes the lead permanently unrecoverable. An endpoint whose contract is "retry exactly the failed rows" cannot ship over that. `ReapStrandedEvents` moves them to `failed` (only where `result_record_id IS NULL`, or a retry double-writes).
   - **Sequential, never a worker pool:** item k must see item k-1's write, or two rows sharing an email create two contacts. Ledger timeout (15s) deliberately EXCEEDS the write timeout (7s): the ledger context is armed first and every bookkeeping call hangs against it, so inverting them makes post-write `FinishEvent` fail systematically and strands rows.
   - **207, never a flat 200 over failures**, plus `X-Lead-Batch-Failed` (Make/Zapier sail past any 2xx without reading a body) and a ledger row for every refusal, so the envelope and the delivery log cannot tell different stories.
   - Deferred as pre-existing platform work, NOT batch's to fix: DB connection-pool limits, the 5s shutdown grace (the reaper makes a hard kill recoverable), and routing machine traffic around the Cloudflare Pages proxy.
9. **Lead → deal — DONE (verified live).** Per-source "also create a deal", executed at `ingest.go` after the contact write and before `FinishEvent`, on `ictx`. Decisions (6 designs + 3 adversarial lenses):
   - **"Pipeline + stage" was a plan-level fiction: there is no Pipeline entity in this codebase.** No `pipelines` table, no `pipeline_id` column; `PipelineStage` is a flat per-org list ordered by `position`. The option is therefore STAGE ONLY, and the UI never says the word "pipeline".
   - **A deal only when the lead CREATES a contact.** The deciding fact is that `CountCreatedToday` counts contact creates, so on the matched path the daily cap bounds *nothing* — a leaked key replaying one existing contact could mint unbounded deals under a cap of 10. Gating on create makes `deals ≤ created contacts ≤ daily_cap` true by inheritance rather than by a second mechanism nobody maintains; the rotation ticket and test-lead convergence come free on the same branch. **The cost is real and was the strongest argument against**: a returning customer's second submission — often the highest-intent lead a form receives — produces no deal. So it is DISCLOSED, not hidden: the delivery says "matched an existing contact, so no deal was created", the card says it in words, and the plan's own workflow recipe (`contact_created` + `trigger.source` → `create_record`) is the documented escape hatch for orgs wanting one per submission.
   - **Stage is validated at save AND re-resolved at ingest.** Save-time alone is provably insufficient: stage deletion is a SOFT delete and `pipelineStageUseCase.Delete` asks nothing about who references the stage, so an admin restructuring their pipeline months later leaves an id that still satisfies the FK. The result is worse than an invisible deal — `ObjectKanban` buckets an unknown stage into its FIRST column for display while the DB disagrees, so the card looks correctly filed and reports grouped by stage orphan it. A forecast that is wrong and looks healthy. Ingest falls back to the first live stage and says so on the delivery; `viewOf` badges `deal_stage_missing`. **Polarity is split deliberately**: a stage lookup that ERRORS keeps the configured stage (a result we could not obtain is not evidence the stage is dead — the `routing.go` liveness rule), while a stage confirmed GONE falls back loudly.
   - **Won/lost stages refused at save.** Deal creation does not derive `is_won`/`is_lost`/`closed_at` — only `ChangeStage` does — so a deal started in "Closed Won" sits in the won column reporting `is_won=false`: wrong in the board and wrong in the forecast, in opposite directions.
   - **Config lives in the EXISTING `config` JSONB, under a `deal` key.** It is the one storage option that adds no new failure mode to the capture path: `config` has been inside `CREATE TABLE lead_sources` since the table existed, so unlike every ALTER-added column it cannot be missing where the table is present — which is the exact hazard that forced `owner_pool` and `consent` to be unmapped. **Zero boot guards, zero migrations — the first L2 item with no prod schema risk.** `UpdateSource` now `Omit`s the column and `SetDealConfig` writes one key via `jsonb_set`, so L3/L5 source-kind config can sit beside it without two admins deleting each other's settings.
   - **NEVER for a test lead.** A test contact is inert; a test deal is COUNTED — forecast total, P9 revenue — and unlike the contact it has no convergence mechanism (no `ProviderEventID`, so each click is an honest separate delivery), and the plan already refused an `is_test` marker on customer schema. Five clicks would leave five real deals with nothing marking them synthetic. The test panel reports it through `uncovered`, the field that exists to stop a green result reading as "everything works".
   - **A failed deal never costs the lead** — the contact is already written, attributed and owned by then, so every failure is a `slog` line + a caller warning + a delivery note, and never a `failEvent`. Same polarity as attribution seeding and the consent write.
   - **The title template uses a CLOSED token vocabulary** (`full_name`/`first_name`/`last_name`/`email`/`company`/`source_name`/`date`), rendered locally rather than through automation's `InterpolateTemplate` (which is built around `EvalContext` — the wrong type to construct to format one string). Closed for two reasons, the second load-bearing: an open vocabulary over the org allowlist would make a deal title an FLS READ primitive, and **a deal title outlives erasure** (`RedactForRecord` is scalar equality on `result_record_id` and touches only `integration_events`; `contactUseCase.Delete` is a soft delete so the `ON DELETE SET NULL` never fires). Every token is a reviewable decision about what may survive a deletion request — pinned by a test that fails if the list changes. Unknown tokens are rejected at SAVE (fail closed, an admin is watching) and render BLANK at ingest (a kanban card reading `{{first_name}} — Website Form` looks like a broken product to the customer's own sales team). A non-blank ladder — rendered → source name → "New lead" — exists because `dealAdapter.create` 400s an empty title, which would turn a naming mistake into a lost deal.
   - **Ledger records the deal in `event.Note`, not a second column.** A `result_deal_id` was designed and rejected: the reaper releases stranded rows only `WHERE result_record_id IS NULL`, so persisting a second id risks the exact permanent-409 trap the reaper exists to prevent — and the re-run-duplicate argument for it evaporates under the create-branch gate (a retry finds the contact it just wrote, returns `updated`, and creates nothing). The deal id reaches the CALLER instead, on `captureResponse`/`batchItemResult`, where it is actionable at integration time.
   - Deferred, explicitly: erasure reaching `deals.title`; a referential guard on stage deletion; deals on the matched path.
   - **Three adjacent defects found and FIXED in the same change, all pre-existing:**
     1. **`batch_enroll_automation` was decoratively broken in BOTH directions.** `sourceRequest` had no such field, so gin silently dropped the key the UI sent; `SetBatchEnrollAutomation` had zero callers; `viewOf` never hydrated it, so a GET always returned `false`. **STATUS: DONE (verified live).** Decisions (6 recon areas + 4 adversarial lenses):
- **Platform facts verified against Google's raw docs, and two bit us.** (a) The Samples page spells the field **`Google_key`** (capital G) on test payloads and `google_key` on the production one, *on the same site*. An exact-case map lookup would 401 every advertiser's first test and the integration would be dead on arrival while every one of our tests passed — so the envelope decodes through a typed struct (encoding/json matches field names case-insensitively as a fallback), pinned by a test that uses Google's payloads verbatim, capital G included. (b) The "does the test button echo the CONFIGURED key or a placeholder?" question was the one gamble in the design; it is now settled *for* hard verification by triangulating three official pages (the Samples page's two test payloads carry the page's own demo key, not the Testing page's `"testkey"`).
- **`is_test` runs as `TestOriginProvider` — the one payload-derived origin the pipeline permits — but its IDENTITY is COERCED, not asserted.** Google's sample data (`test@example.com`, a documented dummy phone) has done its one job by then (exercising the real field map, with unmapped questions already quarantined and observable), so email is rewritten to the synthetic `crm-test-{source}@lead-test.invalid` and phone is stripped. Refusing traffic while flagged answers before the body is read, so the lead leaves **no ledger row at all**; when Google's finite retry budget expires it is gone without a trace, and the source can never demonstrate recovery because nothing is ever attempted. A transient blip would brick the source until a human noticed, with L6 alerting unbuilt. So `IsLive()` now excludes only `disabled`, flagged sources keep accepting, and `TouchSourceUsed` un-flips on any real success — self-healing. It also un-bricks the bearer/batch **recovery** endpoints, which a gate would have 401d with a message telling the integrator to rotate a healthy credential.

Status transitions moved to a targeted `SetSourceStatus`.
- **Rate limiting answers 503, never 429**, on this route only — same reasoning as the cap.
- **`public_token` + `google_key_hash` are UNMAPPED** (the owner_pool rule): ALTER-added columns whose failed boot guard would otherwise 500 every capture request in every org. ## L4 — Web-to-lead form embeds

**Problem:** The long tail: customers' own websites. Also the first **browser-origin** caller, which the engine-wide credentialed-CORS allowlist currently forbids.

**Build:**
1. Source kind `form_embed`: per-form `public_token`, per-form field list, allowed-origins list, anti-spam config.
2. Public endpoint `POST /api/capture/forms/:public_token`. **CORS design task (explicit review item):** the global `cors.New` middleware allows only the app's own origins with credentials — verified: it 403-aborts disallowed *browser* Origins (server-to-server posts without Origin are unaffected, so L1/L3/L5 are safe). Capture-form routes need per-source dynamic origin allowlists: wrap the global CORS handler with a path-prefix skipper for `/api/capture/forms/` and implement route-level CORS (echo allowed origin, **no credentials**, `POST, OPTIONS`) from the source's allowlist. A mistake here either breaks embeds or opens credentialed CORS to arbitrary origins — dedicated security pass required.
3. Anti-spam: honeypot field (non-empty → accept-and-quarantine), per-IP + per-source rate limits (fail-closed), optional Cloudflare Turnstile (free tier confirmed; per-source keys).
4. Snippet builder UI: copy-paste HTML `<form>` + tiny JS that auto-captures `location.href` (UTMs ride along), referrer, posts JSON; success/error hooks. No iframe hosting in v1.

**Done when:** the snippet on an external origin submits successfully from an allowed origin and is blocked from a non-allowlisted one; UTMs land on the contact; honeypot submissions quarantine silently; burst posting throttles.

**STATUS: DONE (verified live).** Decisions (6 recon areas; the CORS pass was the design-blocking one):
- **The global CORS handler had to be SKIPPED, not widened — and the difference is the whole security story.** Verified in the vendored `gin-contrib/cors@v1.7.7`: `applyCors` returns immediately when there is no `Origin` (so L1/L3 server-to-server is untouched), and `AbortWithStatus(403)`s any origin not in the three-entry allowlist **before any route handler runs** — so "just add CORS to the route" is dead on arrival. Widening looks tempting because `AllowOriginWithContextFunc` exists, but it is **additive-only, receives no path context, and cannot change credentials**: `generateNormalHeaders` bakes `Access-Control-Allow-Credentials: true` into the header maps at CONSTRUCTION from the single global config. So admitting a customer origin through it would tell that origin it may send the victim's ambient cookies to this API **and read the response** — a cross-origin read of authenticated CRM data on every route in the app. Echoing an arbitrary origin is precisely the dangerous form; a bare `*` cannot do this, because browsers reject `*` with credentials. `main.go` therefore skips the global handler for `/api/capture/forms` and the route carries its own credentials-free one.
- **The route handler is HAND-ROLLED rather than a second `cors.New`.** That constructor **panics** when its config has no origins and no origin function — exactly the shape a per-request DB-driven allowlist has — and a panic there is a boot failure that takes the whole API down.
- **An origin allowlist is a nuisance filter, not an authentication check, and that sentence is in the code AND the admin UI.** curl sends no Origin and sails straight through, by design — it is the same request shape L1/L3 integrators use legitimately. What bounds a script is the rate limiters, the daily cap and Turnstile. The comment exists because the predictable future mistake is someone removing the bot check "because we have an origin allowlist".
- **Empty allowlist DENIES every browser origin** — and that is the state of every newly created form, so it is the single most likely way this feature could silently fail open. The card warns loudly until an origin is added.
- **`allowed_origins` is its own column, NOT a key in the `config` blob** where the form's other settings live. That parser is documented never to fail (missing/junk/null all mean "feature off"), which is right for a naming template and wrong for a security list: an allowlist needs **unreadable** to be distinguishable from **empty**, because the two must have opposite outcomes. Burying that choice inside a parser that cannot fail is how it gets made by accident.
- **The DECLARED field list does the job the credential does elsewhere.** Undeclared keys are DROPPED, not quarantined: quarantined keys are stored verbatim and then sampled by the mapping UI's observed-keys query, so keeping them would let anyone who viewed the page source inject arbitrary strings into the admin's mapping picker.
- **Every accepted submission gets one CONSTANT response body.** An early version emitted `consent_recorded` on the success path only — which let a bot detect a honeypot catch by that key's ABSENCE and adapt. The integration test caught it. Anything an anonymous caller could correlate with the outcome belongs on the ledger.
- **Nothing on this route feeds the source's error badge except an Ingest 5xx and the no-record guard.** Every other failure is forgeable by a stranger, so counting them would let anyone flip a healthy form into an alarm. **Consequence recorded now rather than discovered at L6: `status='error'` is near-decorative for `form_embed`, and L6 alerting must not lean on it for this kind.**
- **The wire idempotency key is IGNORED here.** On an anonymous route a caller could pre-claim keys, and the replay switch re-runs failed/quarantined rows in place — so a pre-claimed row would be a way to bind a later real lead to a stranger's payload.
- **`context` is rebuilt server-side from the two keys we read**, never passed through, or an anonymous poster could write arbitrary JSON onto the ledger row whose whole value is saying what actually happened.
- **Turnstile's failure polarity is OURS to choose — Cloudflare documents none.** Neither pole is acceptable (fail-open makes an outage a spam window; fail-closed makes it a lead-loss window), so an unreachable verify accepts-and-quarantines, the third branch the daily cap already established. A stale or already-spent token is the ONE case the visitor is told about, because it is overwhelmingly a real person who left the tab open and a silent 200 would hand them a thank-you and lose their enquiry. Keys are the customer's own: the free tier is 20 widgets per account, so ours would cap the platform at twenty forms.
- **Live cross-route bug found and fixed:** `public_token` is one namespace across kinds and neither capture route checked the kind. A form token — public by construction — POSTed at the Google route resolved a real source, failed the key check it could never pass, and planted an alarming "the webhook key Google sent did not match" row in that org's ledger. Anyone who viewed source could manufacture those. Both routes now gate on kind.
- Deferred: `TRUSTED_PROXIES` is unset, so `ClientIP()` is the edge and per-IP limiting is coarse — **setting it is a prod precondition for this kind**, since a browser endpoint is the first one where per-IP actually matters. Also deferred: subdomain wildcards in the allowlist (the naive `HasSuffix` shape matches `evilexample.com`), and a per-event replay button.

**L6.1 — Unified health signal + status-flip notifications: DONE (verified + reviewed, uncommitted).** Built: failure counting extended to the channels that never had it (`countSourceFailure` is now the one entry point; the L1 capture API counts 5xx-class ingest errors + the no-record guard, batch counts a wholly-failed run as ONE envelope); `BumpConnectionFailure`/`EaseConnectionHealth`/`MarkConnectionSynced` giving connections the counter and the `last_synced_at` their columns already declared; `health.go` (the `Notifier` + `HealthAudience` ports, the `HealthReporter` with per-band dedupe, a fleet circuit breaker and a drained `Stop()`, and the copy table); `repository.IntegrationAudienceReader` (the org-admin recipient query, which did not exist in any form); the `integration_health` entry in `domain.NotificationEventTypes`; main.go wiring + shutdown drain. **Design panel + 4 judge lenses + 4 adversarial hazard angles (12 agents, 16 HIGH hazards), and the findings changed the design rather than decorating it:**
- **The plan's own Done-when was unreachable, and the design that "satisfied" it would have reopened L3's hole.** A rotated-away google_key fails `hmac.Equal` and 401s at `google.go:245-253` **before any ingest**, and L3 deliberately never counts pre-auth failures because they are attacker-forgeable. So the counter can never see the flagship scenario, and a source whose key was rotated away keeps reading green while every real lead Google sends is rejected. Detection has to come from the mismatch ledger row instead: a **`SourceKeyMismatch` alert that writes no status, touches no counter, and is debounced to once per source per day**. It ships despite being attacker-triggerable because **the threat model is the REMEDY, not the trigger** — this alert asks "check your key" (harmless if manufactured), whereas a red badge asks "disable the source", which is why *that* signal must stay unforgeable. The copy names both possible causes because we genuinely cannot tell them apart.
- **NO new columns — and that is the load-bearing decision, not a scope cut.** The natural design is a `health_notified_status`/`health_notified_at` watermark pair for notification dedupe. **Eight of the sixteen HIGH hazards traced to those two columns and to nothing else:** a mapped watermark whose boot guard failed 500s every capture request in every org; the same columns added inside `integration_connections`' `CREATE TABLE IF NOT EXISTS` guard are a silent no-op on prod, so the OAuth callback 500s **after** the single-use state is burned and connecting a page becomes permanently impossible; folding them into `TouchSourceUsed` regresses L3/L5 behaviour that works *today* (last_used_at freezes fleet-wide, `error` can never self-heal) rather than merely disabling the new feature; and because a watermark commits with the UPDATE while dispatch happens later, **a rolling deploy loses the escalation permanently — while deploys are themselves among the likeliest causes of the failures being escalated**. The existing atomic statements already hand back a genuine once-only edge (`IncrementSourceFailure` has returned `flipped` since L3 and it was being thrown into a log line); `TouchSourceUsed` gained a `prev` CTE so it reports `healed` the same way. **A statement that is wrong fails in every environment at build or test time; an ALTER-added column fails silently on prod alone.** Dedupe therefore lives in memory, which is bounded, degrades toward *noisier* rather than *silent*, and cannot be switched off by a Redis outage.
- **`degraded` is added for CONNECTIONS and refused for SOURCES**, and the asymmetry is justified by code state rather than taste. On connections it is already a declared const, already inside the claim index's `status IN (…)` predicate, and already mapped in the FE `STATUS_VARIANT` — and it finally gets a producer for the hole it was declared for: **429/5xx are classified Retryable, so they never flipped anything, and a sustained Graph outage exhausted the 3-attempt budget, dropped every delivery, and left the card reading `connected`** — silent lead loss behind a green badge. On `lead_sources.status` the same value would need two proven atomic statements widened, would render as a **blue, healthy-looking pill** on the old FE bundle (the `Record<LeadSourceStatus,…>` is exhaustive with no fallback), would mislabel the toggle as "Enable" (secretly a clear-the-counter button), and would strand **zombie rows on rollback that no shipped code path can leave** — permanently disarming the error badge for exactly the sources that were mid-failure.
- **Retryable failures are CAPPED at `degraded` and can never reach `error`** — that cap is what keeps the `error` copy true. `error` means "your credential stopped working, reconnect"; letting a Graph rate-limit reach the same band would tell an admin to redo OAuth because Facebook was throttling us. A genuine credential death still escalates *out* of degraded (pinned by test).
- **A fleet circuit breaker, because extending counting to the busiest channel is what turns a latent storm into a real one.** Every kind now counts an Ingest 5xx, so one RecordService or DB fault flips every source in every org at once — and the fan-out would mail every admin on the platform a message **blaming their own integration for our outage**, while hammering the database that is already failing. Above 5 distinct orgs in 15 minutes the reporter logs once and stops notifying. The badges still go red, because that part is true; only the accusation is withheld.
- **`last_synced_at` is stamped at terminal delivery success and nowhere else.** The tempting site is the connection heal, but that fires right after `FetchLead` and **before** form resolution — so a connection quarantining 100% of its deliveries would report a fresh sync time forever, the most confident green the UI has over a pipe producing no records.
- **The owner is added to the recipient set explicitly.** The owner role holds no `role_permissions` rows *by design* (it bypasses capability checks so an empty permission table cannot lock an owner out), so the natural query — "roles granting `integrations.manage`" — silently excludes the one person guaranteed to care. That omission passes every test written against a seeded org.
- **Copy rules, each because the obvious sentence is false here:** never "stopped receiving leads" (`error` is a live, self-healing badge); never render the counter as a rate (it counts unbroken RUNS and cannot honestly carry one); never point CONNECTION copy at the delivery log (connection-side failures leave `source_id` NULL and the only events route filters on `source_id`, so that log *structurally* cannot contain them until L6.3). All three are pinned by tests that assert the forbidden strings are ABSENT.
- Deliberately **not** solved here: the rate-vs-run gap (a source failing 50% never flips). The leaky-bucket fix turns heal-in-one-success into heal-in-sixteen, leaving a low-volume source badged red for over a week and breaking the existing "a real success self-heals the badge" assertion. Partial failure is shown as **data in the event log (L6.3)**, not as an alarm. `form_embed` alerting stays near-decorative exactly as L4 documented — this slice does not pretend otherwise.
- Found but NOT fixed here (filed for L6.3): the L1 capture daily-cap path 429s **before** Ingest and writes no ledger row at all, unlike google_ads and form_embed which both store evidence — so a capped source is invisible in its own delivery log. Also: the digest's 100-row cap has no per-type quota, so a flapping source can crowd a member's workflow notifications out of their daily email.

**L6.2 — The delivery log you can act on: DONE (verified + reviewed, uncommitted).** Built: `ListEventsFiltered` (keyset, source/connection/status/unresolved filters) + `GetEvent` + `RequeueEventForRetry` + `SourceLabels`; `retry.go` (the `classifyRetry` verdict + `eventView`/`eventPage`); `event_handlers.go` (`GET /api/integrations/events`, `POST /api/integrations/events/:event_id/retry`); an org keyset index; `RependEvent` org-scoped; the two L6.1 defects. FE: `DeliveryLogSection` page + route, `listEventLog`/`retryEvent`/`RetryRefusedError`, `useEventLog`/`useRetryEvent`. **3 designs + 4 adversarial hazard angles (7 agents, 22 HIGH hazards), and the hazards cut the feature down rather than decorating it:**
- **RETRY IS ASYNC-ONLY, and that is the finding.** Seven of the 22 HIGH hazards were ONE hazard wearing different hats, all in the path where a button re-runs a SYNC delivery from its stored payload: the reaper fails a running replay as "the server restarted" inside its 10-minute grace and, by flipping the row back to `failed`, re-opens the guard so a second run starts on top of the first; the guard cannot see a concurrent Idempotency-Key re-run at all, because that path re-runs in place and never writes `processing` to the DB; `result_record_id` — the anti-double-write guard everything leans on — is written by the very call that can fail AFTER the record exists, so it is NULL exactly when a lead HAS been written; and `FinishEvent`'s wholesale Save writes NULL back over a result another run just committed, re-arming the button on a delivery that already produced a contact. **All four are unreachable if stored payloads are never replayed on demand** — and sync already has a recovery path that is safe by construction (the sender re-sends with the same Idempotency-Key and re-enters the replay switch, where those invariants already live). A button would duplicate that path with none of its protection, and could not work at all for the two pre-auth shapes (a Google key-mismatch row stores a REDACTED envelope; a form rejection stores nested unauthenticated body). So retry means one thing: hand a provider delivery back to the worker to **re-fetch**. It covers the case worth the most — a Facebook lead quarantined because its form was not enabled, recovered the moment the admin enables it.
- **The retry button is a callerless contact write, so it takes the test-lead guard.** Ingest writes with no `domain.Caller`, so OLS/FLS never run; without a re-check, `integrations.manage` alone becomes a standing contact-write primitive for whoever holds it. `authorizeTarget` re-authorizes the CLICKING admin's own create+edit with their real caller — the second button in this position, taking the same guard as the first.
- **The backfill exclusion is structural, not a marker.** An import row carries a `connection_id` like any provider delivery and would otherwise qualify — and re-running it through the webhook door drops `DeliveryBackfill`, the ONLY thing suppressing automation on a historical import: one click could enrol months of old leads into every `contact_created` workflow and mail all of them. The discriminator is `source_id`, which backfill stamps at insert and a pre-ingest webhook failure never has, because rows written before any marker existed would be indistinguishable — and **"we cannot tell" must not resolve to "send the emails"**. The same predicate also excludes webhook rows that failed during the write, which the worker has already retried automatically.
- **A liveness re-check BEFORE the row is touched.** The commonest retryable row is a form-not-enabled quarantine; if the admin has not enabled it yet, re-queueing achieves nothing and is not free — the claim increments `attempts`, and after `maxWebhookAttempts` the row is terminal. A few hopeful clicks would **permanently destroy the only recovery path those leads have**, while the ledger note still says "enable it in Integrations settings".
- **The shipped per-source route is untouched, byte for byte.** The frontend coerces an unexpected envelope to `[]` rather than erroring, so repointing it at the new nested shape would render a full ledger as "No deliveries yet" for anyone whose bundle and backend are one deploy apart — and an admin debugging a live source would conclude nothing had ever been sent and rotate a healthy key. The new envelope lives on the new route only.
- **The new index needs a NEW NAME.** `idx_integration_events_org_created` is already taken, and `CREATE INDEX IF NOT EXISTS` matches on NAME ONLY and never compares columns — so reusing it would have been a silent no-op and the ledger would seq-scan with nothing to show for it. An index rather than a column, so a failed guard degrades to a slow query instead of breaking `FinishEvent`.

And a `processed` row with no record is surfaced as a LOST LEAD rather than "nothing to retry".
- **The two L6.1 defects, both fixed.** (a) The L1 capture daily-cap path 429'd **before** Ingest and wrote no ledger row at all, unlike google_ads and form_embed — so a capped source went silent in its own log, which reads as "the integrator stopped sending" rather than "we refused them". Evidence is now written first; the 429 is unchanged, because unlike Google a bearer-authenticated integrator retries. (b) The digest had no per-type quota, and the crowding was **permanent loss**: the fetch is oldest-first and capped at 100, so a flapping source filled the window and the workflow notifications behind it stayed pending until they aged past the 7-day lookback — never emailed, never explained. **L6.5a — Bulk-delete ledger erasure: DONE (verified + mutation-proven, uncommitted).** The single-contact delete has redacted the lead ledger since L2.7, and `BulkAction "delete"` called nothing at all — so a customer honouring a data-protection request **one person at a time was covered, and the same customer honouring it over a LIST was not**. The raw payload, the capture context and the consent envelope all survived, for every subject in the request. Exactly the case with the most people in it erased none of them.
- **The sharp part was WHICH ids to erase, and the obvious answer was a data-destruction primitive.** `BulkDeleteByIDs` carries a row-level write scope (`applyWriteScopeFromCtx`), so an own-scoped caller's request silently skips the records they do not own — it returned only a COUNT, so "requested" and "deleted" were indistinguishable to the caller. Redacting `input.ContactIDs` would therefore have let any member with bulk access **destroy ledger evidence for contacts they were never allowed to touch**, gated by nothing but org membership. Fixed at the source: `BulkDeleteByIDs` now RETURNS the ids it actually deleted (`clause.Returning`, one implementer, one caller, no fakes), `Affected` is derived from that set, and erasure keys off it. Pinned by a test where the scope lets one of three through.
- **A batch statement, not a loop.** `RedactForRecords` is one UPDATE with `result_record_id IN ?`; a bulk delete can carry hundreds of ids, and a round trip each is how erasure becomes the slowest thing in the request, then gets made asynchronous, then becomes unreliable. Empty input is an explicit no-op — `IN ()` is a Postgres syntax error, and an empty slice must never degrade to "erase everything".
- **Best-effort, matching the single path**: the contacts are already gone, so failing the request would report a deletion that DID happen as having failed — the one answer that makes a deletion request impossible to honour at all. Logged with the count.
- **Verified:** build + vet clean; Docker-backed `usecase`, `repository`, `integrations` and `domain` suites green. **Mutation-proven twice** — removing the erasure call fails two tests, and keying it off the requested ids instead of the deleted ids fails the own-scope guard with its intended message. A parity test asserts the batch form erases byte-identically to the singular form, so the two shapes of one rule cannot drift.
- Still open in L6.5: rows with `result_record_id IS NULL` (every failed/quarantined delivery) hold the subject's verbatim payload and are unreachable by contact-keyed erasure by construction; the 90-day prune that would be their backstop does not exist and needs the first advisory-locked job in the codebase.

**L6.3 - Per-connection diagnose: DONE (verified, uncommitted).** "Reconnect this account" was the only thing the product could say when a connection stopped producing leads, and it is wrong at least as often as it is right. Built: `CheckSubscription` on the `Provider` interface (+ Facebook impl reading `GET /{page}/subscribed_apps`), `POST /api/integrations/connections/:id/diagnose`, `CountLiveFormSources`, and a checklist panel on the connection card.
- **The subscription READ probe is the new capability, and it covers the layer that was invisible.** `Subscribe` only ever WRITES, so "subscribed" was a belief recorded at connect time and never re-tested — and a subscription can lapse without touching our token (an admin removing the app in the page's Business Settings, a permission review, a Meta-side change). Until now that page looked perfectly healthy and simply never delivered another lead. `HealthCheck` already existed for Facebook with **zero callers**; it now has one.
- **The last layer is on OUR side, not the provider's.** A connection can pass every remote check and still produce nothing because no form was ever enabled — the commonest "it isn't working" of all, and the only one the admin fixes on that screen.
- **The server sends NO prose — only keys and statuses.** That is the house posture, and here it is also a security control: a provider error embeds the request URL and a page token rides in that query string (the L5.2 token-leak fix exists because exactly that string reached a rendered field). A fixed vocabulary cannot leak a secret.
- **`unknown` is never collapsed into `fail`.** "We could not ask" and "the answer was no" lead to different actions, and conflating them sends an admin to redo OAuth over a provider outage. It is also how the LAM case is reported honestly: a permission error on the subscription read can mean the provider was unreachable OR that Leads Access was never granted, and the copy names both rather than asserting either. `skipped` is rendered rather than hidden, because an omitted row reads as "fine".
- A provider with no probe reports `unknown`, not `fail` — every future adapter embeds `UnimplementedProvider`, and the alternative would make each one read as broken on the day it ships.
- **Verified:** build + vet + full `-short` clean; tsc + 959 FE tests. Live verification of the Facebook layers is gated on Meta track M.

**L6.4b - Workspace-deletion teardown: DONE (verified + mutation-proven, uncommitted).** Deleting a workspace touched no integrations table at all: sealed provider credentials stayed encrypted at rest with nothing left that could reach a UI to remove them (deletion evicts every member), the page CLAIM went on blocking every other workspace from a page the customer could no longer release, and the async backlog kept writing into the deleted org. Built: `PurgeService` + `ListClaimedConnections`/`PurgeConnectionSecrets`/`DisableSourcesForOrg`/`PurgeOAuthArtifactsForOrg`/`DeletedOrgsNeedingPurge`; an `IntegrationsPurger` port; backfill cancellation; a repair arm on the retention sweep. **3-angle adversarial pass, and it caught a cross-tenant break plus a factually wrong justification:**
- **The teardown reached into ANOTHER tenant.** `Provider.Disconnect` is `DELETE /{account}/subscribed_apps` - it detaches our app from the ACCOUNT and knows nothing about connections, orgs or claims. Sweeping every connection the org ever had would unsubscribe pages it disconnected long ago and **another workspace has since connected**: that workspace's card would still read "connected, receiving leads" while Meta silently stopped delivering, and nothing would alert, because the health signal counts fetch failures and there would be no fetches. The unsubscribe set is now only the accounts this org still HOLDS the claim on, re-checked against the live claim immediately before each call. Credential DESTRUCTION keeps the opposite scope (every historical row) - blanking a secret harms nobody; calling a provider on another tenant's behalf does.
- **The purge ran BEFORE the org soft-delete on a justification that was simply false.** The comment claimed the teardown's queries join `organizations`; none of the four does. So the ordering bought nothing and risked everything: a `SoftDeleteOrganization` that then failed would leave a workspace **still alive and still signed into** with its credentials destroyed, its claims released and every source disabled - reported to the owner as "delete failed". It now runs last, after the evictions.
- **Connections are purged BEFORE sources**, the reverse of the intuitive order. `GetConnection` is soft-delete-scoped, so once the connection is gone the async worker fails the delivery WITHOUT fetching. Disabling sources first leaves the connection live, so the worker fetches the lead from the provider - pulling a real person's details - and only then quarantines it, **writing fresh personal data into the workspace the customer just deleted**.
- **An in-flight backfill outlived the deletion by up to 15 minutes**, holding a `*LeadSource` and opened credentials in memory and re-checking nothing - so blanking the database could not stop it importing real people, and with enrolment on it could mail them. The in-flight map now stores the run's `CancelFunc` instead of a bare `true`.
- **A repair arm, because best-effort with no retry surface is just permanent failure.** A deleted workspace can never be deleted again, so a transient deadlock would have stranded a sealed credential behind one log line. The retention sweep now also finds soft-deleted orgs still holding a credential or an enabled source and re-runs the teardown - which additionally **repairs the past**.
- Also: `context.WithoutCancel` so closing the tab cannot tear the teardown in half; OAuth custody rows purged FIRST so no half-finished handshake can complete behind it; the counts recorded on the `workspace.deleted` audit event, because the org's own ledger is unreachable afterwards; and the delete dialog now names the external effects instead of only "this can't be undone".
- **Verified:** build + vet + full `-short` clean; Docker-backed `integrations` + `usecase` green; tsc + 959 FE tests. **Mutation-proven on both scopes.**
- Deferred: the daily digest has no org-liveness join, so a deleted workspace's owner can still receive one digest the next morning.

**L6.5b — Ledger retention + erasure repair: DONE (verified + mutation-proven, uncommitted).** Closes the half of L6.5 that contact-keyed erasure cannot reach. Built: `redacted_at` (migration 000051 + boot guard + the harness list) and a `idx_integration_events_prunable` partial index; `StartLedgerPrune` (advisory-locked, batched, bounded) and `PruneExpiredPayloads`; `quarantined_fields` added to the SHIPPED redactors. **4-angle adversarial pass, and it overturned four things in the first design — two of which would have made the product WORSE:**
- **Blanking `context` would have destroyed the retry path.** The quarantined-lead recovery flow resolves which form a delivery belongs to from `context->>'form_id'`; with it gone the pre-flight answers "this lead's form is not enabled" **forever, on a form the admin had just enabled**, for every row. Context is now PROJECTED — provider routing ids (`form_id`, `page_id`) stay, `gclid`/`utm_*`/`referrer_url` go. Pinned by a test that re-runs `classifyRetry` on a pruned row.
- **`result_record_id IS NOT NULL` does NOT mean "erasable on request".** Contact deletion calls the redactor **best-effort and only logs on failure**, and a second delete cannot re-trigger it (the soft-delete finds no live row and returns before reaching the redactor) — so a missed redaction is PERMANENT, and scoping the sweep to orphans would have excluded precisely the rows where erasure was promised and did not happen. A second arm repairs them: `result_record_id IS NOT NULL AND NOT EXISTS (live contact)`, applying the same write contact deletion would have, consent tombstone included, with **no age gate** because the erasure was already due. This is what makes the word "backstop" true rather than aspirational.
- **`quarantined_fields` holds subject-supplied VALUES and was redacted by NOTHING** — not by the sweep, and not by the already-shipped contact-keyed erasure. It is where the free text a visitor typed into an unmapped field lands, while the UI told the admin "what the person supplied is gone". Fixed in the sweep AND in `RedactForRecord`/`RedactForRecords`: an existing, live compliance gap this slice happened to surface.
- **The consent invariant is DEFENDED, not assumed.** An orphan "cannot" carry an envelope (consent is written only after a record exists) — but `FinishEvent`'s wholesale Save can null a committed `result_record_id` while consent, being unmapped, survives, which `retry.go` already documents. The sweep tombstones consent residue: writes nothing when the invariant holds, erases when it does not.
- **`redacted_at` is a real column and it is worth the schema risk** — unusual for this project, where L6.1 added none on purpose. It is needed twice over: the sweep RETAINS some context keys, so a payload-shape idempotency guard would re-match forever and the batching loop would never terminate; and an operator must be able to tell a redacted delivery from one that never stored anything, or retention is indistinguishable from a bug. UNMAPPED (`gorm:"-"`) like consent, because `FinishEvent`'s wholesale Save would otherwise write it back to NULL; a guard that never ran degrades to "the sweep errors and logs" — it is in no capture-path SELECT, so it cannot 500 a lead.
- **Redacts, never deletes**: what must go is the personal data, not the fact a delivery happened — the row and its status are the ops record and the answer to "did anything arrive at all". Also excluded: non-terminal rows and rows with no `processed_at` (their payload is the input a worker is about to use, and `FinishEvent` could write it straight back), and the `incomplete` class, whose payload the delivery log tells an admin to open and re-send.
- **Shipped copy corrected**: "There is no automatic deletion yet" was true when consent shipped and is not any more. It now names the rule that applies rather than denying any rule exists — and the consent promise survives intact, because consent only ever exists on rows this sweep exempts.
- **Verified:** build + vet + full `-short` clean; Docker-backed `integrations`/`usecase`/`repository` green; tsc + 957 FE tests. **Mutation-proven on all three HIGH guards** — blanking context, dropping arm 2, and dropping `quarantined_fields` each fail their intended test. Idempotency/termination is asserted directly (a second sweep must find zero), as is advisory-lock exclusivity AND release.
- **Follow-up shipped in the same slice:** `redacted_at` is now on the wire and the delivery log renders it — an empty payload with no explanation is indistinguishable from a delivery that stored nothing, which is what a bug looks like, so the panel says the lead was erased and when.
- Deferred, explicitly: a per-org retention window (the 90 days is fixed and stated in the UI, not configurable); and a notification on first sweep activation.

**L6.4a — Member-offboarding routing cascade: DONE (verified + mutation-proven, uncommitted).** Pulled forward out of L6.4 because it was a **live data bug**, not missing work.

- **The bug: removal repaired one half of a two-part binding.** `RevokeUserGrants` pruned the departing member from every source's `owner_pool` and left `default_owner_id` pointing straight at them — and `resolveOwner`'s non-pooled branch stamps that column **unchecked**, deliberately (L1 semantics must not silently change for sources that never opted into rotation). So every source where the leaver was the single owner went on assigning new contacts to a user with **no `org_users` row at all**. Own-scoped reps cannot see records owned by someone else, so those leads were invisible and nobody triaged them — verbatim the failure this whole platform opens by describing. The single-owner shape is also the *commonest* one, since most sources have an owner and no rotation.
- **The 409 copy already promised the fix.** `RoutingSourceNames` matches `default_owner_id = ? OR owner_pool @> ?`, and `ReassignmentRequiredError.Error()` renders "They will also be removed from the lead rotation on: …" over that list. The message named sources the repair never touched.
- **Why `RevokeUserGrants` had no test at all** is the whole reason this survived: nothing covered it, and a test written for the pool alone would have passed throughout the bug's entire life. The new tests assert **both bindings together against the shipped migrations**, plus org-scoping (the same person can be in two workspaces), plus a **drift guard** that runs `RoutingSourceNames` before and after and requires everything the disclosure named to actually have been repaired. Mutation-proven: reverting the new statement fails two tests with their intended messages.
- **Cleared to NULL, not reassigned to the removal's transfer target.** That target answers "who inherits the records they already own", which is a different question from "who gets the next lead this form receives"; quietly reusing one answer for the other is how a rep discovers they own a channel nobody told them about. Unowned is also the deliberately louder state — L2.5 already settled that stamping an inactive member is *worse* than leaving a lead unassigned, because it looks handled — and the unowned path shouts in four places already. Same suspension asymmetry as the pool: suspension does **not** clear it, because it is reversible and the column is how the rep gets their pipe back on reinstate.
- **The disclosure never reached the admin, on either path, and both are fixed.** (a) `routingSources` was fetched unconditionally but only attached to the 409 raised inside `if ownedContacts+ownedDeals+ownedCustom > 0` — so a recently-added SDR who is on the rotation but has closed nothing yet, **the commonest offboarding there is**, was removed in total silence. `RemoveMember` now returns a `*domain.MemberRemoved` and the 200 carries `routing_sources_cleared`, so the warning ships on the no-409 path too. (b) Even when the 409 *did* fire, `routing_sources` was on no wire — it existed only inside the free-text message, which the handler's own comment forbids the SPA from parsing. It now has its own key. Also recorded in the `member.removed` admin audit event, which is what answers "why did this form stop assigning leads?" three weeks later.
- **Ride-along fix:** the frontend `ReassignmentRequiredError` had been silently dropping `owned.custom` since U6.3 — the backend sent it, the parser ignored it, and the dialog under-reported the impact in the very place the admin decides what happens to the data.
- **Verified:** build + vet + full `-short` suite clean; **Docker-backed `repository`, `integrations` and `usecase` suites green**; tsc clean; 948 FE tests (4 new, covering both disclosure paths and a negative control — a banner that appears on every removal is noise, and noise is how the real warning stops being read). The post-removal notice is mutation-proven too.

**L7.1 — Instagram, and the diagnose layer that never worked: DONE (verified + mutation-proven, uncommitted).** Built: `platform`/`is_organic` on the Graph lead node, carried into the delivery context by the shared `rawLeadFromFB`; a `withPlacementFallback` ladder; `FacebookProvider.CheckSubscription`; the relabel across the user-visible strings; and the org-wide delivery log's missing context block. **4-angle adversarial review (20 agents, 16 findings raised, 2 survived) — and the interesting part is what the skeptics KILLED:**
- **The headline finding was that `CheckSubscription` never asked Graph for `subscribed_fields`** — it is the only Graph read in the file that sends no `fields` param — and that a nil field list on a matched app id was being returned as a definite `false`, so every healthy page would diagnose as "not subscribed" and every admin would be told to redo OAuth. Two independent angles raised it as HIGH. **The skeptics refuted it on the platform fact**: Meta's Page/subscribed_apps reference lists `subscribed_fields` as a *Default* field of that edge, with a captured bare-GET response to match, so the nil branch is unreachable against Graph as it behaves today. The projection was added anyway, because the residual the refutation itself names is real: the verdict hangs entirely on Meta continuing to include a field by default, and the cost of that changing is not a missing label but every working connection reporting broken. **The nil case is now an ERROR (→ `unknown`), not `false`** — "our app is attached and Graph told us nothing about what it is subscribed to" is a third state, and folding it into "no" is the precise mistake this layer exists to prevent.
- **The ladder exists because a lead must never be lost to a reporting field.** An unrecognised entry in `fields` fails the WHOLE Graph node request, so adding `platform` to answer "are Instagram leads arriving?" would otherwise have put every Facebook *and* Instagram lead behind it. It retries narrow only on a PERMANENT failure: a throttle or an outage says nothing about the field list, and retrying narrower against one would burn a second request and learn nothing. Backfill shares the ladder for the opposite reason — no live lead is at risk there, but a split failure would leave historical import permanently broken while the webhook path kept working, which reads as "backfill is buggy" rather than "this field does not exist".
- **`FacebookProvider.CheckSubscription` did not exist.** L6.3 shipped the interface method, the route and the panel; Facebook inherited `UnimplementedProvider`, so the *Delivery subscription* layer — the one that slice was written for, the lapse a token probe structurally cannot see — has answered `unknown` for the only shipped provider since the day it landed. It now matches on OUR app id, which is defensive rather than crowd-control: the page token is app-scoped so the list is realistically 0–1 entries, but a row belonging to another agency's CRM must never read as our health.
- **Both surviving findings were about tests and reach, not logic.** A skeptic *mutation-proved* that the backfill placement test stayed green with Backfill reverted to the narrow field list — it pinned the fallback rung but not the request WIDTH, so a future narrowing would ship placement-less historical imports with CI silent. And the org-wide delivery log never rendered `event.context` at all, which matters because a quarantined Meta delivery has `source_id NULL` and therefore appears on NO source's page: the one key the slice exists to expose was unreachable for exactly the deliveries an admin opens that page to understand. Both fixed; the backfill test now asserts the width and the cursor, and is mutation-proven against the skeptic's own revert.
- **Verified:** build + vet + full `-short` clean; tsc + 960 FE tests. The app-id match and the backfill request width are each mutation-proven.

**L7.2a — the legacy upsert's data loss, fixed where it lives: DONE (verified + mutation-proven, uncommitted).** `POST /api/webhooks/inbound/:org_token` keeps its route, its wire contract and its trigger payload byte-for-byte; only the raw-SQL upsert's three defects are repaired. **Zero payload change is the whole point** — see L7.2b for why re-routing the write is not a refactor.
- **Every write error was discarded, and the 200 was worse than uninformative.** A failed INSERT still returned `{"status":"accepted","contact_id":"<uuid>"}` with a uuid that was never inserted — and then fired a `contact_created` workflow against that phantom id, so runs referenced a record that does not exist. The duplicate-email race is the shape a busy integrator actually hits; it now re-reads and applies the delivery to the winning row, reporting `contact_updated`, which is what actually happened.
- **`custom_fields` was REPLACED wholesale** with only the current payload's unknown keys, so a delivery carrying one custom field destroyed every other custom field on that contact — including values a human had typed. Now one `jsonb ||` statement rather than a read-modify-write, so concurrent deliveries for the same contact cannot lose each other's keys.
- **A failed SELECT fell through to the INSERT branch**, so a transient database blip created a duplicate contact instead of updating the existing one. A query failure is now an error and never conflated with "no such contact".
- **A numeric phone lost the whole lead.** The raw JSON value went straight into a varchar column; the insert failed and the error was discarded. Scalars are coerced — it cannot regress stored data, because the alternative outcome was no row at all — while the trigger payload keeps the caller's raw value so no workflow sees a different phone than it does today. `first_name`/`last_name` deliberately keep their legacy string-assert (a non-string becomes `""`): there is no lost lead to recover there, so widening it would be a product change wearing a bug fix's clothes.
- **The test that mattered most was the one that first proved nothing.** A wall-clock concurrency test passed with the fix fully reverted — on a fast machine the deliveries serialize and no conflict ever occurs. The race is now FORCED: a BEFORE INSERT trigger parks the handler on an advisory lock the test holds, the test commits a competitor on another connection, then releases. (The first attempt had the trigger write the competitor itself — which fails, because an INSERT that violates a unique index rolls the whole statement back and takes the competitor with it.) Mutation-proven: restoring the discarded-error behaviour fails it with exactly the phantom-id symptom.

- **Verified:** build + vet clean; 6 new Docker-backed tests against real Postgres, plus the forced-race test; two independent mutations each fail their intended test.

**L7.2b — the legacy endpoint keeps its write and gains the platform: DONE (verified + mutation-proven, uncommitted).** Two passes re-scoped this slice away from the plan's specification, and the second overturned the first.

**Pass 1 (4 angles, 46 hazards) killed the shim.** The plan's own acceptance test could not see its own failure mode — a corpus asserting status codes and response bodies passes byte-identically through every hazard, because the interface that changes is the TRIGGER PAYLOAD. The legacy handler emits the caller's submitted JSON; RecordService emits a projection of the stored row; the two share only `id`. `{{contact.company}}` vanishes, `first_name` becomes synthesized from the email local-part (all 25 shipped starter templates interpolate it), a numeric phone becomes `"5.551234e+06"`, `custom_fields` becomes the union of every past delivery, attribution OVERWRITES a caller's own `lead_source`, and `owner_user_id`/`company_id`/`tags` appear. The engine renders an unresolved path as `""` and answers false for every operator except `is_empty` on an absent one, so each of those silently rewrites live customer workflows.

**Pass 2 (3 judges + a mechanics check) killed the migration too, and found the cheaper answer.** Asked to choose between *accept the payload change behind an opt-in* and *preserve the payload with a compatibility emitter*, the judges split three ways and converged on something neither option had: **the write-move is what causes the harm, and the benefit does not require it.**
- **The pre-flight can be an inventory, but never a verdict.** Token extraction is exact (it can reuse the engine's own regex); deciding breakage is not — and the in-repo precedent already gets it wrong: `validator.go`'s template scan inspects only top-level string params and never descends into `update_record.updates[]`, `find_records.filters[]` or `create_record.fields[]`.
- **One failure mode loses leads outright and NO pre-flight can see it.** The first ingest seeds text-typed attribution field defs; `validateCustomValuesOnly` then type-checks them, so a caller who has posted a numeric `utm_campaign` for years starts getting a 400 with no event at all. The pre-flight scans workflows; this is about payload shapes, and there is no legacy ledger to sample them from — that absence being the whole problem.
- **Preserving the payload was refuted by its own advocate.** A compatibility emitter hands workflows `{{contact.company}}` and `{{contact.custom_fields.budget}}` values that exist NOWHERE in the database, so a workflow that `update_record`s or `find_records` off them acts on phantom data — something today's legacy path cannot do, because what was submitted *was* what was written.
- **And `owner_user_id` is a COLUMN, not a payload key.** Stamping an owner changes who can SEE the lead without touching one template or condition. The complaint this whole platform opens with — own-scoped reps never see legacy leads — is fixable with no payload risk at all.

**So the write and the trigger payload are untouched, byte for byte, and the endpoint borrows only what a workflow cannot observe**, through `automation.LeadCapture` (declared in the consumer with primitive signatures — the `LeadLedgerRedactor` precedent — so neither package imports the other) implemented by `integrations.LegacyCapture`:
- a **delivery ledger** row, opened BEFORE the write so a lead whose write failed is still recoverable, and closed after — legacy traffic finally appears in the delivery log at all;
- **owner routing** through the existing three-rung ladder, stamped on the create branch only;
- the **health signal** — the same atomic increment and one-transition announcement `countSourceFailure` makes, safe to call here because the HMAC has already been verified, so a stranger who reads an org token off a proxy log still cannot flip the source red.

**Zero new columns.** `lead_sources.kind` is a plain `VARCHAR` with no CHECK constraint — a deliberate decision recorded in `models.go` precisely so adding a kind is a new VALUE rather than a schema change — so the whole ALTER-column-fails-silently-on-prod hazard class that L6.1 called its load-bearing decision does not apply here. The only schema artifact is a partial unique index.

**Review (4 angles, 20 agents; 13 confirmed findings deduplicating to 4 defects) — and the HIGH was mine:**
- **`BeginDelivery` burned a rotation ticket on every delivery.** The ticket is CONSUMING and I resolved it before the create/update decision was known, so every resubmission spent a rep's turn on a lead that gets no owner. That is verbatim what `ingest.go:563` calls "this feature's worst bug", reintroduced in the most resubmission-heavy channel in the product: with a steady one-new-in-three mix and a three-person rotation, every new lead lands on the same rep while the Integrations page shows a healthy three-way split — and the fairness tests still pass, because they exercise consecutive tickets. Fixed by splitting the port so `ResolveOwner` is called from the create branch and nowhere else, and **mutation-proven**: restoring the eager call fails the new test with 5 burned turns.
- **The routing note was assigned to an already-inserted struct and never persisted.** It is the ONLY disclosure that a lead landed unowned, because this endpoint's 200 body is frozen and there is no warnings channel to say it on. Now written together with the owner in one targeted statement.
- **`ON CONFLICT DO NOTHING` had no constraint to fire on**, so two concurrent first deliveries would create duplicate sources that the new delete guard then made permanently undeletable. The partial unique index that makes the one-row-per-org assumption true now exists — boot guard, migration 000052, and the harness list, all three, or the tests pass against a schema that lacks it.
- **Daily cap, deal creation, field mapping and update policy saved successfully and did nothing** — the exact "switch that silently does nothing" hazard the status and delete guards were added to prevent, and worse, because these look like they took effect. Rejected server-side and hidden in the UI. `default_owner_id`/`owner_pool` deliberately stay: routing is the one capability this path really honours.

Deliberately NOT borrowed: **attribution**. A legacy payload carries no `context` envelope, so the only attribution available is `lead_source`/`lead_source_detail` — which collides with a key legacy senders already use, is not reportable without seeding field definitions, and seeding those is precisely the mechanism that turns a previously-accepted numeric value into a 400 that loses the lead.

**Verified:** build + vet + full `-short` clean; Docker-backed `automation` and `integrations` suites green; tsc + 966 FE tests. Two mutations each fail their intended test (the rotation ticket, and the create-only owner stamp).

**Still open, unchanged by this slice:** nothing here moves legacy traffic onto mapping, the allowlist, RecordService validation or the deal option, and the raw-SQL writer lives on. The full payload cutover remains available as its own decision later — now decoupled from owner routing, which was the coupling that made both original options expensive.

A numeric `phone` becomes `"5.551234e+06"` via `%v`. `custom_fields` becomes the merged union of every past delivery, and attribution OVERWRITES a caller's own `lead_source`. New keys (`owner_user_id`, `company_id`, `tags`) appear.
- **The engine makes every one of those silent.** `InterpolateTemplate` renders an unresolved path as `""` with a `slog.Warn`; `evaluateLeaf` returns **false for every operator except `is_empty`** when a path is absent. So a key appearing flips `neq`/`not_contains` from false to true, and a key vanishing flips them the other way — no error, no failed run, no trace.
- **Two paths become hard failures where legacy returned 200:** the first ingest SEEDS text-typed attribution field defs, after which `validateCustomValuesOnly` type-checks them — so a caller who has posted `{"utm_campaign": 1234}` for two years starts getting a 400 and NO event at all. And quarantine/policy short-circuits emit no event where legacy always emitted one.
- **Therefore: opt-in per org, not a silent cutover.** The shape to build is a migration an admin chooses, gated behind a pre-flight that scans that org's workflow definitions for `contact.company` / `contact.custom_fields.*` / `trigger.source` references and names the affected workflows before anything moves; the legacy path stays live for un-migrated orgs. Other decisions already taken by the pre-mortem: the route STAYS in `internal/automation` (it owns the token lookup, the HMAC, the 100/min bucket and all three response shapes — moving it would re-implement the wire); the write core arrives via a port injected from main, because `internal/usecase` imports `internal/automation`, so an in-package test cannot import `integrations` without an import cycle (the real-pipeline E2E has to live on the integrations side, or in an external `automation_test` package); all four existing tests build `&Handler{}` literals, so a new port field silently defaults to nil; and `passthrough_custom_fields` needs its own security pass (it re-admits the blacklist, is unbounded and append-only under an uncapped source, and its key names reach the mapping UI's observed-keys picker).

**L7.3 — the global HMAC kill switch, and a panel that described the wrong thing: DONE (verified, uncommitted).**
- **`WEBHOOK_SKIP_SIGNATURE` was a bare `os.Getenv` read with no environment gate** — the only production read of it in the backend — so setting it anywhere disabled signature verification for EVERY org's public endpoint at once, with no log line, no UI indication, and no record that the variable exists (it is in no config struct, no `BindEnv`, and neither `.env.example`). A production deploy that inherited it from a dev shell handed anyone who could read an org token — it travels in the URL, and the URL is written to the access log — contact-creation and workflow-firing in that workspace. Now gated on the `APP_ENV` **exact allowlist**, the `debugTokensEnabled` precedent: a `!= "production"` check fails OPEN because APP_ENV is unset on prod today, which is exactly how account-takeover debug tokens once shipped. The handler's `appEnv` ZERO VALUE means production, so a handler built without `SetAppEnv` can never be talked out of verifying.
- The four existing tests that set the flag now declare `appEnv: "test"`; without it the skip is inert and every unsigned fixture delivery 401s. Pinned by a table asserting `""`, `production`, `staging`, `Development` and `TEST` all REFUSE (the case-sensitivity cases are the allowlist's point), a positive control that the hatch still works in `development`/`test`, and a third that a dev environment ALONE is not permission.
- **The builder's webhook panel opened with a false sentence.** "POST to this URL to trigger the workflow" — but the endpoint emits `contact_created`/`contact_updated` and workflow lookup matches on the trigger TYPE, so a workflow saved with that Webhook trigger is the one kind an inbound POST can never start. The panel is also the only place the URL and secret are shown, so the misdescription lived exactly where someone was setting the integration up. It now says which workflows deliveries actually run, and points at Settings → Integrations for per-source keys, mapping, routing and a delivery log. The Payload Fields note also stops claiming `company` maps to a contact field — it is read and passed to workflows, but contacts hold a company RELATION and it has never been stored.
- **Verified:** build + vet clean; the gate's tests are Docker-backed; tsc + 653 workflow FE tests.

**L7.4 — LinkedIn as a relay, on purpose: DONE (verified, uncommitted).** `docs/linkedin-lead-gen-forms.md` in the house recipe style, plus a `RelayRecipes` card on the Integrations page.
- **Prose on the page, not a link to the doc.** The app has no docs route and nothing serves the repo's `docs/` directory — `lib/docs.ts` supports one operator-configured `VITE_DOCS_URL` and renders nothing when unset — so the plan's "linked from the Integrations page" would have shipped a 404. Inlining provider instructions is the house pattern already (`GoogleAdsSetupCard`, `FacebookFormCard`, `SetupRecipe`).
- **LinkedIn is deliberately not native**, and the card says why in one line: Lead Sync API access is gated on a verified company page and a per-app review, and its versions retire on a schedule, so a native adapter is a maintenance commitment someone else can switch off. The relay cannot be revoked.
- The honest-reading section leads with the failure a relay recipe must own: **the relay is a third system that can fail silently.** A paused scenario or an expired connection stops leads arriving, and our health signal only counts deliveries that REACHED us — so a relay that stopped relaying is indistinguishable from a quiet week. The alert belongs in Make/Zapier. It also names the two things people get wrong: send the platform's lead id as `Idempotency-Key` (without it a scenario re-run relays the same person again, and there is nothing to dedupe on), and expect `lead_source` to read `integration:api` — report on `lead_source_detail`, which is why the recipe says to name the source for the channel rather than the tool.

**L7.5 — TikTok, and the framework generalization it forced: DONE (verified + mutation-proven, uncommitted).** The plan said "a new `Provider` adapter on the L5 framework". Research against TikTok's official docs said otherwise, and building it proved the framework was still Facebook-shaped in four places.

**The three platform facts that are capability differences rather than dialect:**
- **The webhook CARRIES the lead.** `GET /lead/get/` has NO lead id parameter — it takes a form and returns one unspecified lead — so there is no by-id read to make. The whole L5.3 pipeline was built on "the webhook carries ids only, so the fetch is where the lead data enters". The fix turned out to be ONE LINE: the worker rebuilt `InboundEvent` without `Raw`, so rehydrating it from the stored delivery lets such a provider implement `FetchLead` as a pure function, with the claim/retry/health machinery around it untouched. `ProviderInfo.CarriesLeadData` declares the intent.
- **The signature is not over the raw bytes.** `X-Open-Signature` is HMAC-SHA256 over an ASCII-ESCAPED form of the payload; TikTok's docs say plainly that hashing the decoded bytes yields a different signature. A byte scan rewrites non-ASCII runes as `\uXXXX` (astral planes as surrogate pairs) without re-marshalling, because re-encoding would reorder keys. Hashing the raw body would have worked for every ASCII payload and failed the first time a lead carried an accented name — a real person's lead, rejected in a way that looks like an attack.
- **Failure arrives as HTTP 200.** Every response carries a `code` that "takes precedence over HTTP status codes". The shared client classifies on status, so without `decodeTikTok` a dead token reads as success and the lead behind it is written from an empty response. Throttles (40016/40100/40132/40133) stay Retryable; `20001` is a documented SUCCESS code and is not treated as failure.

**What was Facebook-shaped and is now generic:** `ProviderInfo` gained `SourceKind`, so `config -> <provider> ->> 'form_id'` resolves a delivery for any adapter (`FindFacebookFormSource`/`EnabledFormIDs` were hardcoded in four places); the seed field map moved onto the `Provider` interface; `EnableForm` dispatches kind, config namespace and seed map by provider; and `WebhookHandler` is per-provider, with the GET handshake mounted only for a provider that HAS one — TikTok registers its callback by API call and never probes the URL, so a handshake route there would publish an endpoint whose only behaviour is 403.

**Zero new columns again.** A source kind is a VALUE in a `VARCHAR` with no CHECK constraint; the only artifact is `tiktok_form`'s own partial unique index, since a partial index cannot be parameterised by the kind it filters on.

**Review (4 angles, 34 agents; 12 confirmed findings over 6 defects) — the two that mattered were both mine, and both were consequences of the fetchless design:**
- **The pipeline REWRITES `raw_payload` as a delivery moves through it.** A quarantined or re-ingested row holds the flat field map, not the `{fields, context}` envelope `ParseWebhook` stored — so an envelope-only reader finds nothing on exactly the SECOND attempt. A retried TikTok lead would have been destroyed, and the failure reported as a dead credential. `FetchLead` now reads both shapes; mutation-proven against the envelope-only version.
- **An unusable delivery was being blamed on the credential.** The worker flips the whole connection to `error` — "reconnect this account" — for any non-retryable fetch error. For a provider whose "fetch" is a read of our own stored row, that would page an admin to redo OAuth over one malformed callback. A new `ErrDeliveryUnusable` sentinel fails the delivery and leaves the health signal alone.
- Also fixed: `CreateSource` guarded two kinds by literal and so let `tiktok_form` through, minting a live bearer key on a kind whose UI hides the key and whose Rotate is refused — now a predicate over `IsKeylessKind`, so the next adapter cannot repeat it. `IsFormSourceConflict` matched only Facebook's index name, so the TikTok enable-form race would have 500'd instead of resolving to the winner. The registration gate now includes the auth URL, because registering a provider whose Connect button lands nowhere is worse than not offering it. And the delete-confirm copy keys on the keyless predicate rather than the `facebook_form` literal.
- A ride-along: `TestConnectFlow_UnknownProvider` used `"tiktok"` as its unregistered provider. It still passed once TikTok shipped — it builds its own empty registry — while no longer testing what its message says. Renamed.

**Deferred, explicitly:** BACKFILL. TikTok's historical export is an async task (`POST /page/lead/task/` both creates and polls) yielding a CSV that expires in 10 minutes, whose columns after `form_name` are the form's own LOCALIZED question labels rather than stable keys — and leads live in three regional silos (`x-lead-region: eu`/`us`/omitted) where the wrong header returns nothing rather than erroring. That is its own slice, and none of it is needed for leads to flow.

**Verified:** build + vet + full `-short` clean; Docker-backed `integrations` and `automation` suites green; tsc + 966 FE tests. The retry-path read and the credential-blame sentinel are each mutation-proven. **LIVE verification is gated on a real TikTok app AND an ad account the connecting user administers — unlike Meta there is no sandbox fallback, because TikTok's does not serve the lead endpoints at all.**

**The pre-build research is retained below, because it is the citable record of why each of the above is shaped the way it is:**

**L7.5 — TikTok research: the API does NOT fit the L5 framework as written.** The plan assumed "new `Provider` adapter on the L5 framework … webhook push with 72h retries → same event-dedupe; pull backfill". Verified against TikTok's official docs (the portal is a JS app; the raw doc content was pulled from `business-api.tiktok.com/gateway/api/doc/client/node/get/`), four of those assumptions are wrong, and the first one is structural:
- **`FetchLead` cannot be implemented.** `GET /open_api/v1.3/lead/get/` takes `lead_source` + `advertiser_id`/`library_id` + `page_id` and **has no `lead_id` parameter**; it returns one lead and the docs never say WHICH. There is no by-id fetch. But the whole L5.3 async pipeline is built on "the webhook carries IDs only, so the fetch is where the lead data enters the system" — the claim-then-`FetchLead` worker has no meaning here. TikTok's webhook carries the ANSWERS INLINE (`changes: [{field, value}]`), so the delivery is already complete at receipt. Supporting it means the processor must accept a provider that needs no fetch, which is a change to `webhook_processor.go`, not to an adapter.
- **The signature is NOT over the raw bytes.** `X-Open-Signature` is `hex_lower(HMAC-SHA256(app_secret, escaped-unicode form of the payload))` — TikTok's docs are explicit: "we generate the signature using an escaped unicode version of the payload… If you just calculate against the decoded bytes, you will end up with a different signature." That contradicts security invariant 1 as written ("HMAC-SHA256 over the **raw body bytes**") and needs a byte-scan that rewrites non-ASCII runes as `\uXXXX` — re-marshalling would reorder keys and fail.
- **Leads live in three separate regional silos.** `x-lead-region: eu` / `us` / omitted are three different datasets; a call without the header "will only receive leads from other countries/regions" — i.e. the wrong header silently returns NOTHING rather than erroring. Backfill must fan out across all three, and the default is the one most likely to look like "no leads yet".
- **Backfill is an async task returning a CSV that expires in 10 minutes**, not a paged JSON GET: `POST /page/lead/task/` (the same endpoint creates AND polls, discriminated by whether `task_id` is sent) → `SUCCEED` → `GET /page/lead/task/download/`. The columns after `form_name` are the form's own **localized question labels**, not stable keys, so the mapping engine's source keys would differ per form and per language. Admin role on the ad account is required at runtime, and lead endpoints are **not available in the sandbox** — so unlike Facebook, there is no fixture-only path to a working integration.
- Smaller, all confirmed: the error envelope is HTTP 200 with a `code` field (`0` ok, `20001` = "partially successful" and must not be treated as failure); throttles are `40016`/`40100`/`40133` while `40102`/`40105`/`40106` mean re-auth; the advertiser access token **never expires and has no refresh token**; revoking it **silently kills the webhook subscription**, so reconnect must re-subscribe; ids are JSON *numbers* in webhooks and *strings* in REST; `page_id`/`form_id`/`id` are three names for one concept; the authorize URL's query params are **not officially documented** (operator-supplied config, not constructed); the lead *download* endpoints sit under the **Test Leads** scope rather than Leads Retrieval; app review is 2–3 business days and a later scope change costs another 2–3.
- **Consequence for the plan:** L7.5 is not "ships only after the framework proves itself on FB" — it is a framework slice (a fetchless provider path) plus an adapter, and its Done-when ("a TikTok test lead lands through the shared pipeline") needs a real ad account with an ADMIN role, since the sandbox does not serve lead endpoints. Full citations are in the research output; nothing was built.

## Verification strategy (every phase)

- **Backend:** `go build` + `go vet` + `go test -short ./...`; DB-touching paths get Docker-gated integration tests (run them in Docker before calling a phase done — a skipping suite is not a passing suite). The ingest pipeline gets table-driven matrices: upsert/dedupe/mapping/quarantine/idempotency-replay/**concurrency (parallel same-email creates)**/**NULL-source webhook dedupe**/**update-policy (fill_blank_only vs overwrite vs create_only)**.
- **Frontend:** `cd crm-frontend && npx vitest run` (never rtk-wrapped) + `rtk npx tsc -b`; Radix portals ⇒ `screen.getBy*`.
- **Live:** local stack (`docker-compose` + seed_local_account.js) + curl fixture scripts per channel (in `crm-backend/internal/integrations/testdata/`: Google Ads payload incl. `is_test`, FB webhook envelope + signed body, capture-API envelopes, the legacy-webhook replay corpus); FB end-to-end via the Lead Ads Testing Tool against an owned page. After each prod deploy, drive one test lead through the capture API.
- **Review:** each phase ends with a multi-agent adversarial review (the A-series pattern) with the security invariants as explicit dimensions; the L4 CORS change and every public endpoint get a dedicated security pass; L0 and L7.4 get regression-focused passes (existing writes / legacy contract).

## Key risks & mitigations

| Risk | Mitigation |
|---|---|
| Meta review timeline (weeks) blocks the flagship | Track M starts week 1; L1 Make/Zapier bridge covers FB leads meanwhile (Make = free tier; Zapier webhooks need a paid plan — documented); all L5 engineering works against owned pages in dev mode |
| Callerless trusted writes = field injection / OLS bypass if any guard slips | Strict schema allowlist + ownership/relation blacklist live **from L1** (not L2); source-save re-authorizes the configuring admin's OLS on the target; the ingest context is built in exactly ONE constructor; review checklist item every phase |
| Upsert-on-email is an update primitive over existing records | `update_policy` default `fill_blank_only`; owner/relations never caller-writable; `create_only` available per source |
| Boot-guard/migration drift (works local, 500s prod) | Every table ships both artifacts in the same commit; each phase's Done-when includes a prod smoke lead |
| Backfill/reconciliation enrollment storms | `WithAutomationSuppressed` default for backfill + recovered leads; enrollment is opt-in with a count confirmation |
| Engine per-minute idempotency swallows rapid distinct updates for the same contact | Delivery-level dedupe lives in `integration_events` (provider event IDs); distinct real updates >1/min behave like every other write path today — documented, not fixed here |
| Redis absent ⇒ rate limits vanish on public write endpoints | Fail **closed** to an in-process bucket + DB daily creation cap (email/automation amplification bounded even with both limiters down) |
| Provider API drift (Graph version retirements, LinkedIn sunsets) | Versions pinned per provider; L6 health probes fail loud; LinkedIn deliberately not native |
| Duplicate contacts from phone-only leads before L2 | L1 API sources may declare email required; FB (phone-heavy) doesn't ship until L5, after L2's phone matching |
| Lost `INTEGRATION_ENC_KEY` = unrecoverable credentials | No JWT_SECRET fallback (fail boot loud), keyring + `key_version`, startup canary decrypt, documented rotation runbook |
