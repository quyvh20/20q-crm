# Auth System Improvement Plan (P10)

Goal: make the authorization system (roles, orgs, permissions) fully dynamic, multi-org-correct, and recovery-complete. Produced from a 14-agent audit (6 subsystem mappers, 3 gap analysts, 3 competing designs + judge, completeness critic) with every load-bearing claim verified against source at file:line.

---

## 1. Verified current state — more exists than expected

**R1 (dynamic RBAC) — backend core already built.**
- A fixed 15-code capability catalog (`internal/domain/role.go:126-159`) stored per-role in `role_permissions`; routes gated by `RequireCapability`/`RequireObjectAccess` from data, not role names (`router.go:79-98`).
- Custom roles are fully CRUD-able per org via `/api/roles` (gated `roles.manage`), with clone-from (copies capabilities + OLS + FLS), name-shadow check, in-use delete protection.
- Five global system-role templates (owner/admin/manager/sales_rep/viewer, `org_id NULL`) seeded idempotently at boot (`repository/seeder.go`); default OLS matrix per org via `EnsureDefaults`.
- OLS/FLS/capabilities/layouts/report-shares all **store** `role_id`; downstream systems are custom-role-clean at the data layer. Default-deny for unknown roles.

**R2 (multi-org) — plumbing exists, UX doesn't.**
- `org_users` composite PK (user_id, org_id) with per-org `role_id` — same user, different role per org works end-to-end today.
- JWT carries `org_id`, `role`, `rid` (role id), `ds` (data scope), `tv` (token_version); AuthMiddleware re-resolves (user, org) membership per request via Redis `session:{uid}:org:{oid}` (5-min TTL) or DB fallback.
- `POST /api/auth/switch-workspace` re-mints the token for a validated membership; sidebar WorkspaceSwitcher exists; single-org users see plain text, not a switcher.

**R3 (recovery) — self-serve core built and well-hardened.**
- 256-bit crypto/rand tokens, SHA-256-hashed at rest, 1h TTL, atomic single-use claim, password policy checked before token consumption, bcrypt(12), reset revokes ALL refresh tokens + bumps `token_version` → kills sessions in **every** org (correct for global accounts).
- Enumeration-safe (constant response, detached email goroutine), 30/min/IP rate limit, good test coverage (`auth_recovery_test.go`, `auth_hardening_test.go`).
- No endpoint anywhere lets an admin see/set/reset a member's password — the "admin can't reset" property the user wants already holds by absence.
- OAuth-only accounts: completing a reset SETS a password (converts to dual-login); Google login auto-links by email.

## 2. Security fixes that can't wait (ship first, independent of the overhaul)

1. **`debug_token` fail-open = account takeover.** `ForgotPassword` returns the raw reset token in the HTTP response whenever `appEnv != "production"` (`auth_usecase.go:1198-1200`). Unset/typo'd `APP_ENV` ⇒ any anonymous caller can take over any account across all its orgs. Fix: explicit allowlist (`development`/`test` only). **Same fix for the invite debug_token** (`workspace_usecase.go:100-103`) — leaks 7-day org-access tokens.
2. **Google OAuth has no state validation.** State is generated but never stored or checked in the callback (`auth_handler.go:362-412`) — login CSRF on the exact fallback path R3 leans on. Fix: 5-min httpOnly SameSite=Lax state cookie + constant-time compare.
3. **Privilege escalation to owner.** `PATCH /workspaces/members/:user_id/role` (gated only `members.manage`) can promote anyone to the owner role (`workspace_usecase.go:153-183`). And **TransferOwnership is broken**: gated `members.manage` only, promotes the target WITHOUT demoting the current owner (mints a second owner), doesn't validate the target is an active member, nil-derefs on role lookup failure (`workspace_usecase.go:211-219`, `router.go:83`). Fix: block `is_owner` targets in UpdateMemberRole/InviteMember; make TransferOwnership owner-only, transactional demote+promote, validate target.
4. **Membership mutations never evict the session cache.** UpdateMemberRole/Suspend/Remove/Reinstate/Transfer (`workspace_usecase.go:153-245`) never delete `session:{uid}:org:{oid}` — a removed/suspended/demoted user keeps their old access for up to 5 minutes. Only password reset evicts today. Fix: evict on every membership mutation; also on role rescope/capability edit (or stop caching `ds` in the session value and derive from the rid-keyed permission cache).
5. **Silent mail degrade.** If `RESEND_API_KEY` is unset, prod falls back to LogMailer which writes full reset links (raw tokens) into durable logs while users get nothing (`main.go:697-709`). Fix: prod boot refuses to start (keyed off the fixed allowlist, with a `MAIL_DISABLED` escape hatch); LogMailer logs user id + hash prefix only; add `MAIL_FROM` config.
6. **No DB uniqueness on role names.** No unique index on `roles(name, org_id)` anywhere; an org-scoped role named 'owner' created outside `role_usecase` grants god-mode (owner check is a name string in 5 places). Fix: partial unique indexes + `CHECK (is_system OR name <> 'owner')` via boot guard.
7. **Forgot-password has no per-email throttle** (only 30/min/IP, fails open without Redis) — email bombing; add 60s per-email cooldown (silent, enumeration-safe) + void prior tokens on new request.

## 3. Target design

### 3.1 Identity re-key: role_id everywhere, names for display only (R1 keystone)

- `domain.Caller` gains `RoleID uuid.UUID` + `IsOwner bool`; `Role string` stays for display/audit.
- **Bump the Redis session cache key to `session:v2:{uid}:org:{oid}`** with value `status:role:tv:ds:rid:isOwner`. Do NOT append to the v1 value — the old parser (`SplitN(val, ":", 4)`) would stuff `ds:rid:isOwner` into dataScope and silently widen own-scope users to all-scope during rollback/mixed-version windows.
- Permission caches (`LoadOrgAccess`/`LoadOrgCapabilities`/`LoadOrgFieldAccess`) re-key from `r.name` to `r.id` — zero data migration (grant tables already store role_id; only the cache loaders JOIN to name).
- **Layout resolver re-keys too**: `LoadOrgLayoutRoleMap` currently resolves slug→roleName→layoutID (`object_layout_repository.go:52-72`).
- All owner god-mode sites become `caller.IsOwner`: `permission_usecase.go:91/142/184/204` AND the workspace_usecase name checks (`:39,:159,:164,:190,:213,:226`).
- Bridge release: authorize by RoleID first, fall back to name with `WARN authz_name_fallback`; delete the fallback after prod shows zero hits. CI grep gate at the end: no role-name comparisons outside seeder/templates/tests.
- Note the in-process 60s permission cache is single-instance (`permission_usecase.go:24-69`); acceptable while Railway runs one replica — document it, revisit with Redis pub/sub if replicas ever scale.

### 3.2 Permission catalog: vocabulary in code, grants in data, language in metadata

- `AllCapabilities` stays the compile-time enforcement vocabulary. Add `CapabilityInfo{Code, Label, Description, Group, Sensitive}` metadata: groups = People / Permissions & setup / Working with records / Automation & AI / Oversight. Sensitive ⚠ chips on `roles.manage`, `members.manage`, **`workflows.manage` and `workflows.run_any`** (executors are effectively org-wide write + email + outbound HTTP until §3.5 lands), `org.settings`, `data.export`, `billing.manage`.
- `GET /api/roles/catalog` (any member) — labels/descriptions for UIs.
- `GET /api/roles/options` (any member) — `[{id, name, description, is_system, is_owner, data_scope}]`. Unblocks the Share-dialog 403 (`ReportShareDialog.tsx:41-54` currently dies wholesale when `GET /api/roles` 403s) and feeds every role dropdown. Full grants payload stays behind `roles.manage`.
- New capability `analytics.view` (gates AI forecast/analytics), seeded to admin+manager **with an idempotent boot backfill for existing prod role_permissions rows**; accept (and release-note) that pre-existing cloned roles won't have it.
- **Escalation guards**: (1) `is_owner` roles unassignable outside the fixed TransferOwnership; (2) assigning/inviting into a role holding `roles.manage` or `members.manage` requires the caller to hold `roles.manage`; (3) ratify in catalog copy that `roles.manage` is top-of-chain (equivalent to admin) — decide separately whether granting `roles.manage` into a role requires owner (open decision §5).

### 3.3 Schema changes (ALL as idempotent boot guards in `cmd/server/main.go` — golang-migrate is dead on prod; mirror into numbered migrations for dev parity only)

- `roles`: `is_owner BOOL DEFAULT FALSE` (backfill from name+is_system), `description TEXT DEFAULT ''`, `template_key VARCHAR(40)` (backfill = name for system rows), `seeded_from_role_id UUID REFERENCES roles ON DELETE SET NULL`. Partial unique indexes `(name) WHERE org_id IS NULL` and `(org_id, name) WHERE org_id IS NOT NULL`; one-owner-per-org index; `CHECK (NOT is_owner OR template_key='owner')`; `CHECK (is_system OR name <> 'owner') NOT VALID` → pre-count offenders → VALIDATE. Integrity guards log-and-skip on existing bad data, never crash boot.
- `users`: `default_org_id UUID REFERENCES organizations ON DELETE SET NULL` (validated as ACTIVE membership at every use; invalid ⇒ treat as NULL and clear). Email: lowercase backfill with collision pre-check, non-unique `idx_users_email_lower` first, promote to UNIQUE only after prod shows zero offenders.
- `password_reset_tokens`: `initiated_by UUID` (admin-sent-link audit); boot-time DELETE of rows expired >30 days.
- `invitations`: `resent_at`, `revoked_at`; new tokens use `generateSecureToken` (256-bit) instead of `uuid.New()` (~122 bits); validator accepts both during the 7-day overlap.
- Explicitly unchanged: `org_users`, `role_permissions`, `object_permissions`, `field_permissions`, layouts, `report_shares`, `user_groups`. **Refresh tokens stay user-scoped** (ratified: org binding lives in the 2h JWT + per-request membership re-resolution — org-bound refresh tokens would fragment the rotate-with-successor theft-detection chains).
- Code-only: `ListOrgsByUserID` gains `ORDER BY joined_at ASC, org_id ASC` (today `orgUsers[0]` is nondeterministic DB order and it's the default org at login/refresh/OAuth).

### 3.4 AI subsystem convergence (biggest R1 violation today)

The AI package enforces a parallel hardcoded RBAC (switches on the five role names in `ai/tools.go:248`, `ai/command_center.go:461-1021`, `ai/intent_router.go`) and **bypasses OLS/FLS entirely** — a custom role falls into the default branch with the full tool list and org-wide access.

- Tool list derives from the caller's capability set + OLS map; data access goes through RecordService with the real Caller on context (OLS/FLS/own-scope bind AI exactly like REST); own-scope forks key on `caller.DataScope`; forecast/analytics gate on `analytics.view`; system-prompt persona templated from `{data_scope, capabilities}`.
- Bridge release runs both authorizers and logs divergence on prod traffic before deleting the name-switch. **Enumerate expected divergences up front** — e.g. today AI hard-denies sales_rep deal edits (`command_center.go:876`) but OLS grants edit(own), so convergence WIDENS that path by design; accept it explicitly rather than expecting byte-for-byte equivalence.

### 3.5 Automation execution actor model (missing from every naive design)

Workflow executors write via direct SQL bypassing RecordService (e.g. `automation/executor_update_record.go:316`) — skipping OLS, FLS, own-scope, and the P5a audit trail. `workflows.manage` is therefore a de-facto org-wide-write capability: an own-scope role with it escalates past its OLS by authoring a workflow.

- Decide the actor model (open decision §5): **run-as-creator** (workflow actions carry the author's Caller through RecordService — least surprise, blocks the escalation) vs **run-as-system** (explicit system actor, audit-attributed, documented as privileged). Recommendation: run-as-creator for record writes; system actor only for engine housekeeping.
- Inbound webhooks (`POST /api/webhooks/inbound/:org_token`, public, org-token-auth'd) execute as system today — fold into the same decision; confirm token rotation/revocation story.
- Delete the run_now nil-capChecker name fallback (`automation/run_now.go:146-170`); constructor panics on nil checker.
- Until this lands, `workflows.manage` carries the ⚠ chip and the docs say why.

### 3.6 R2 — multi-org login flow

- **Contract**: `LoginInput` gains optional `org_id` (programmatic clients); `AuthResponse` gains `active_org_id` + `default_org_id`. Server-side selection chain: explicit `org_id` (validated ACTIVE, else 403) → valid `default_org_id` (ACTIVE) → single ACTIVE membership → deterministic first ACTIVE. **Filter to active memberships throughout** — today a suspended-first user gets a token minted into an org that rejects them.
- **Zero-membership state**: login with no active orgs currently mints a token bound to `uuid.Nil` and the user 403s everywhere with no UX. Return an explicit "no workspace" response; SPA shows a dead-end page ("You're not a member of any workspace — ask for an invite").
- **Chooser** (`/choose-workspace`): shown iff ACTIVE workspaces > 1 && no valid default. Full-page card list (org name, your role, member count; suspended memberships rendered disabled). Default-checked **"Make this my default"**; selecting calls existing `POST /api/auth/switch-workspace` with new `set_default: bool`. Default also settable from a star in WorkspaceSwitcher and Settings→Profile. Single-org and defaulted users never see it.
- **Switch hygiene ships before the chooser**: SwitchWorkspace threads real `requestMeta` into `createRefreshToken` (today mints blank device rows) and **revokes the presented refresh token** (today every switch orphans a live 7-day credential).
- **Refresh fails closed**: `org_id` not matching an active membership returns `409 {code:"ORG_UNAVAILABLE", workspaces:[...]}` → SPA routes to chooser ("You no longer have access to Acme"). Delete the silent `orgUsers[0]` fallback (today a cookie-only refresh can silently flip a multi-org user to another org). `clearAuth` splits so `active_workspace_id` survives failed refreshes; durable memory is server-side `default_org_id`.
- **OAuth chooser branch**: GoogleCallback binds an org server-side and redirects with the access token in the URL. Apply the same selection chain server-side; carry a `needs_chooser` flag on the redirect; move the access token out of the query string (fragment or one-time code) while touching this.
- `saveAuth` trusts `active_org_id` from the response instead of inferring from localStorage/array order (kills the UI-shows-org-A/token-is-org-B class).

### 3.7 R3 — recovery completion

Self-serve core kept verbatim. Added:

1. **Admin "Send reset link"**: `POST /api/workspaces/members/:user_id/send-reset-link`, gated `members.manage`, target must be an org member; reuses ForgotPassword internals by user ID; **never returns the token in any env**; stamps `initiated_by`; audits `password.reset_link_sent_by_admin` in the sending org + a user-level event the target can see; shares the per-email cooldown **plus a per-target daily cap** (any org's admin can reach a shared user — prevent cross-org harassment). MembersList button with copy: *"We'll email {user} a reset link. You won't see or set their password — their account spans workspaces."*
2. **Case-insensitive email end to end**: `GetUserByEmail` AND `GetOrgUserByEmail` (invite dup-check — currently bypassable by casing) → `LOWER()` compare; normalize on Register/Invite/Login/Forgot writes; functional index; dupe pre-check runs EARLY (before P2), not just before the unique-index promote. This is the #1 real-world "reset email never arrived" cause.
3. **Invite flow rebuilt**: `POST /invitations/:id/resend` (fresh 256-bit token) + `DELETE /invitations/:id` (revoke) + pending-invitations panel. AcceptInvite becomes one transaction and **UPSERTs/reinstates** the org_users row (today it blind-inserts against possible soft-deleted tombstones and has no already-member check). New non-OAuth users get a set-password step (invite-token-authorized) or "Continue with Google" — today invited users are created PASSWORDLESS with no way in but independently discovering forgot-password.
4. **Stranded passwordless users** (already created by the old AcceptInvite): login page branch — "This account exists but has no password — send a reset link?" Also covers Register-with-pending-invite (409 duplicate today).
5. **RemoveMember cleanup**: actually execute the reassignment strategy (today it's accepted and ignored — records stay assigned to the ex-member), and revoke user-targeted report_shares/record_shares/group memberships so a later re-invite doesn't silently restore access.
6. Hygiene: ResetPassword's alert+audit detached to `context.Background()`+timeout (the documented fire-and-forget lesson); InviteMember email send likewise detached (today synchronous on request ctx, 500s after creating the row); ResetPasswordPage strips the token via `history.replaceState`; send failures write a `reset_email_failed` auth_event; reset audit events attributed at user level (org NULL), not the legacy signup org.
7. Role deletion transaction also deletes `report_shares WHERE target_type='role'` (mirror for group hard-delete) + one-time orphan sweep; role edits bust the layout cache alongside the permission cache.

### 3.8 R1 admin UX (non-technical admins)

- **New-role wizard** killing the zero-OLS trap (new roles start fully denied on every object today, invisibly): Step 1 — start from a template or duplicate an existing role ("start from blank" behind a secondary link with a *"blank roles can't see any data"* banner); server-side `seed_from` template (default viewer) copies capabilities+OLS+FLS, recorded in `seeded_from_role_id`. Step 2 — grouped plain-language capability toggles with ⚠ chips. Step 3 — OLS grid + data-scope radio ("Only records they own / All records").
- **New-object seeding**: `EnsureDefaults` seeds custom roles via `seeded_from_role_id`/`template_key` (today it seeds ONLY the five system roles by hardcoded name — every new object leaves custom roles at zero); ObjectsManager nudges *"3 roles have no access to Invoices"*.
- **Dynamic assignment everywhere**: MembersList `ROLE_OPTIONS` (`MembersList.tsx:6`) and InviteMemberModal `ROLES` (`InviteMemberModal.tsx:9-13`) drop hardcoded arrays for `/api/roles/options`; owner roles render disabled with "Transfer ownership instead". **Member APIs accept/return `role_id`**, not names.
- **Delete-with-reassign**: "12 people have this role — move them to: [picker]", transactional.
- **Duplicate & customize** with optional `reassign_members: true` — the mitigation for immutable system templates (org can't tune its own 'Manager' in place; see open decision §5).
- **One `usePermissions()` hook** over extended `GET /api/auth/capabilities` returning `{capabilities[], data_scope, role_id, role_name, is_owner}`. The five hardcoded frontend gates switch: `WorkspaceSettingsPage.tsx:12` → `can('members.invite')`; `MembersList.tsx:18` → `can('members.manage')`; `ConversationLogPage.tsx:31` → `can('members.manage')`; `RunNowModal.tsx:50` → `can('workflows.run_any')` + creator allowance; `AssigneeSelector.tsx:15` → `dataScope === 'own'`. ReportShareDialog → `Promise.allSettled` + `/api/roles/options`.
- **Local dev**: extend `scripts/seed_local_account.js` to seed a second user, a second org, and a multi-org membership — none of chooser/custom-role-assignment/admin-reset-link/share-to-other is locally testable today. Name the sanctioned dev path for grabbing reset/invite links (debug_token under the allowlist) since LogMailer gets redacted.

Exit criteria:
- **R1**: an admin creates, describes, assigns, and shares to a custom role entirely in-app; AI and automation obey the same grants; grep gate green.

- **R3**: reset works case-insensitively with failures visible to support; admins can send links; no endpoint anywhere accepts another user's password.

## 5. Open decisions — RATIFIED 2026-07-07

1. **Per-org materialization of the five system templates** — **DECIDED: skip.** Keep global system-role singletons; Duplicate-&-customize with member reassignment (P6) is the in-place-edit substitute. The `template_key` / `seeded_from_role_id` columns (added in P0) keep the door open if orgs ask later, without doing the risky per-org backfill now.
2. **Automation actor model** — **DECIDED: run-as-creator** for record writes (blocks the workflows.manage escalation; workflow actions carry the author's Caller through RecordService). System actor only for engine housekeeping. Implemented in P8.
3. **`roles.manage` chain-of-trust** — **DECIDED: ratify roles.manage as admin-equivalent (top-of-chain)**, documented in catalog copy; plus the two assignment-time escalation guards (P6). Not the full "never grant what you don't hold" invariant.
4. **`ai.use` capability** — **DECIDED: skip for now**; AI access follows OLS + `analytics.view` once P7 lands. Cheap to add later if a customer wants per-role AI on/off.
5. **Boot refusal vs degrade** when prod mail unconfigured — **DECIDED: refuse to start** (fail closed), with an explicit `MAIL_DISABLED=true` escape hatch for prod-like self-hosts that genuinely run without email. Implemented in P1.

## 6. Answers to the original questions

- **"Admin assigns permissions, nothing hardcoded, customizable roles + default sets"** — the engine already works this way (capabilities in data, custom roles, five seeded templates, default-deny). Perfection = deleting the remaining name-keyed enforcement (owner god-mode strings, AI parallel RBAC, five frontend gates, hardcoded dropdowns) and re-keying runtime identity to role_id so rename/duplicate can never corrupt authorization.
- **"Org chooser with set-as-default"** — plumbing exists (switch-workspace, per-org roles); build the chooser page + `users.default_org_id` + deterministic, ACTIVE-only, fail-closed org selection. Today the default org is literally whatever Postgres returns first.
- **"Admin can't reset passwords — what if the user forgets?"** — correct intuition, and the codebase already agrees: accounts are global (one users row, org_users memberships), no admin password endpoint exists, and self-serve email reset is built and well-hardened (a reset kills sessions in every org). The workarounds to add: admin **"send reset link"** button (email goes to the user; admin never sees anything), Google sign-in as alternate login (exists — fix its CSRF hole), and the rebuilt invite/re-invite flow with a set-password step. Plus the P1 fail-closed fixes so the recovery path can't be silently broken or leak tokens.
